# AETHER Discovery Protocol Design Document

**ステータス**: 設計検討中 (Draft)
**最終更新**: 2026-01-28
**目的**: DHTに代わる、または補完する革新的なピア発見・通信確立プロトコルの設計

---

## 1. 問題の定義

### 1.1 解決すべき本質的な問題

> **「共有秘密を持つ者同士が、第三者に一切情報を漏らさずに、互いを高速に発見し、接続または非同期でメッセージを届ける」**

### 1.2 サブ問題への分解

| サブ問題 | 説明 | 現状の課題 |
|:---|:---|:---|
| **発見問題** | 共有秘密から、相手の「現在地」を知る方法 | DHT/NetDBはIP露出、検索行為が監視可能 |
| **接続問題** | 現在地が分かった後、NAT越えして接続する方法 | STUN/ホールパンチで概ね解決済み |
| **非同期問題** | 相手がオフラインのとき、メッセージを預ける方法 | Mailboxへの依存、信頼問題 |

## 3. 入力仕様 (INPUT)

### 3.1 必須入力

| 入力 | 型 | 説明 | 取得タイミング |
|:---|:---|:---|:---|
| **相手の公開鍵** | `[u8; 32]` | Ed25519公開鍵 | 連絡先交換時 |
| **共有秘密** | `[u8; 32]` | X3DHで確立 or 事前共有 | 初回接続時 |

### 3.2 オプション入力

| 入力 | 型 | 説明 | 取得タイミング |
|:---|:---|:---|:---|
| **Lookup Token** | `[u8; 32]` | DHT検索用（現行案） | 連絡先交換時 |
| **最後の接続IP** | `SocketAddr` | キャッシュ | 前回接続成功時 |
| **タイムスタンプ** | `u64` | 現在時刻（UNIX秒） | 常時 |
| **Mailbox Hint** | `Vec<SocketAddr>` | 既知のMailboxノード | 連絡先交換時 |

## 4. 出力仕様 (OUTPUT)

### 4.1 成功時

| 出力 | 型 | 説明 | 条件 |
|:---|:---|:---|:---|
| **Direct Route** | `SocketAddr` | 相手への直接接続情報 | 相手がオンライン、NAT越え成功 |
| **Relay Route** | `Vec<RelayHop>` | 中継ノード経由の経路 | 直接接続不可、Onion経由 |
| **Mailbox Route** | `Vec<MailboxInfo>` | 非同期配信用Mailbox | 相手がオフライン |

### 4.2 失敗時

| 出力 | 説明 |
|:---|:---|
| `NotFound` | 相手が見つからない（長期オフライン等） |
| `Unreachable` | 見つかったが接続できない（NAT越え失敗等） |
| `Timeout` | 発見プロセスがタイムアウト |

## 5. 制約条件 (CONSTRAINTS)

### 5.1 セキュリティ制約 (MUST)

| ID | 制約 | 説明 | 検証方法 |
|:---|:---|:---|:---|
| **C1** | IP非露出 | 自分のIPがネットワーク全体に公開されない | 通信ログ解析でIPが特定不可 |
| **C2** | 検索内容の秘匿 | 「誰を探しているか」が第三者に分からない | 検索クエリが暗号化/ハッシュ化 |
| **C3** | 通信相手の秘匿 | 「誰と誰が通信しているか」が第三者に分からない | 中継ノードでも送受信者不明 |
| **C5** | Sybil耐性 | 大量の偽ノードによる監視・妨害への耐性 | 攻撃コストが利得を上回る |
| **C6** | 改ざん耐性 | 偽の通信経路を注入されない | 全経路情報に署名検証 |

### 5.2 セキュリティ制約 (SHOULD)

| ID | 制約 | 説明 | 備考 |
|:---|:---|:---|:---|
| **C4** | 参加の秘匿 | AETHERを使っていること自体が分からない | Traffic Shapingで対応中 |

### 5.3 パフォーマンス制約

| ID | 制約 | 目標値 | 理想値 |
|:---|:---|:---|:---|
| **P1** | 発見遅延 | < 500ms | < 200ms |
| **P2** | 発見時通信量 | < 10KB | < 5KB |
| **P3** | 同時接続数 | 100+ | 1000+ |
| **P4** | メモリ使用量 | < 50MB | < 20MB |

### 5.4 可用性制約

| ID | 制約 | 説明 | 対応方針 |
|:---|:---|:---|:---|
| **A1** | オフライン対応 | 相手がオフラインでもメッセージを届けられる | Mailbox |
| **A2** | NAT越え | 双方がNAT配下でも接続できる | STUN + ホールパンチ |
| **A3** | 分散性 | 単一障害点がない | 中央サーバー不要 |
| **A4** | ブートストラップ | 初回起動でも接続開始できる | 最小限のハードコード |

### 5.5 UX制約

| ID | 制約 | 説明 | 目標 |
|:---|:---|:---|:---|
| **U1** | 簡単な連絡先交換 | QRコード1回スキャンで完了 | < 5秒 |
| **U2** | 透明性 | ユーザーは内部動作を意識しない | 全自動 |
| **U3** | 復旧可能性 | デバイス紛失時に復旧可能 | シードフレーズ or バックアップ |

## 6. 既存技術の評価

### 6.1 比較表

| 制約 | DHT | Tor | I2P | F2F | 目標 |
|:---|:---|:---|:---|:---|:---|
| **C1: IP非露出** | ❌ | ✅ | ❌ | ✅ | ✅ |
| **C2: 検索秘匿** | ❌ | ⚠️ | ❌ | ✅ | ✅ |
| **C3: 通信相手秘匿** | ❌ | ✅ | ✅ | ✅ | ✅ |
| **C4: 参加秘匿** | ❌ | ⚠️ | ❌ | ✅ | ⚠️ |
| **C5: Sybil耐性** | ⚠️ | ✅ | ⚠️ | ✅ | ✅ |
| **P1: 低遅延** | ✅ | ❌ | ❌ | ✅ | ✅ |
| **A1: オフライン対応** | ⚠️ | ⚠️ | ⚠️ | ❌ | ✅ |
| **A3: 分散性** | ✅ | ❌ | ✅ | ⚠️ | ✅ |
| **A4: ブートストラップ** | ⚠️ | ⚠️ | ⚠️ | ❌ | ✅ |

### 6.2 各技術の問題点

#### DHT (Kademlia)
- ノードのIPがルーティングテーブルに露出
- 検索クエリが平文（誰が何を探しているか分かる）
- Sybil攻撃に脆弱（コストなくノード生成可能）

#### Tor
- Directory Authority への依存（半中央集権）
- 高遅延（200ms-2000ms）
- 外部への匿名アクセスが主目的（内部通信には過剰）

#### I2P
- NetDB（DHT相当）でFloodfillノードのIPが露出
- 非常に高遅延（500ms-3000ms）
- ネットワーク規模が小さく匿名セットが狭い

#### Friend-to-Friend (F2F)
- ブートストラップ問題（最初の友人をどう作るか）
- ネットワークが分断されやすい
- 新規ユーザーの参加障壁が高い

## 7. 設計空間

### 7.1 発見方式の選択肢

| ID | 方式 | 概要 | 長所 | 短所 |
|:---|:---|:---|:---|:---|
| **D1** | Pull型（DHT） | 能動的に検索 | 高速 | 検索行為が痕跡 |
| **D2** | Push型（Gossip） | 自分を広告 | 受動的 | 遅い、帯域消費 |
| **D3** | Rendezvous型 | 待ち合わせ | 検索不要 | 事前調整必要 |
| **D4** | Direct Cache | 前回情報利用 | 最速 | 初回/IP変更に弱い |
| **D5** | Friend Relay | 友人経由 | IP秘匿 | 最初の友人問題 |

### 7.2 非同期方式の選択肢

| ID | 方式 | 概要 | 長所 | 短所 |
|:---|:---|:---|:---|:---|
| **M1** | Fixed Mailbox | 特定ノードに預ける | シンプル | 単一障害点 |
| **M2** | Distributed Mailbox | 複数ノードに分散 | 耐障害性 | 複雑 |
| **M3** | Epidemic | 全体に広げる | 確実に届く | 帯域、プライバシー |
| **M4** | Store-and-Forward | 経路上で一時保持 | 効率的 | 経路依存 |

## 8. 新プロトコル候補案

### 8.1 案A: Temporal Rendezvous Protocol

**コンセプト**: 時刻と共有秘密から動的にRendezvous Pointを計算

```
Rendezvous_ID = HMAC(SharedSecret, Floor(Timestamp / Interval))

→ 同じ秘密を持つ者同士が、同じ時間帯に同じ「場所」に現れる
→ DHTへの「検索」が不要
→ 時間経過で自動的にRendezvous_IDが変わり痕跡消滅
```

**評価**:
- C1 ✅, C2 ✅, C3 ✅, P1 ⚠️, A1 ❌

### 8.2 案B: Invite-Chain Network

**コンセプト**: 招待制で信頼の連鎖を構築

```
新規ユーザー ← 招待 ← 既存ユーザー ← 招待 ← ... ← 初期ノード

→ ネットワーク参加にはPoW + 招待が必要
→ Sybil攻撃が極めて困難
→ 招待者が中継ノードとしても機能
```

**評価**:
- C1 ✅, C5 ✅, A3 ⚠️, A4 ❌

### 8.3 案C: Hybrid Discovery Protocol

**コンセプト**: 複数方式のフォールバック

```
優先度1: Direct Cache（前回のIP）
    ↓ 失敗
優先度2: Temporal Rendezvous（時刻ベース待ち合わせ）
    ↓ 失敗
優先度3: Lookup Token DHT（暗号化検索）
    ↓ 失敗
優先度4: Mailbox（非同期配信）
```

**評価**:
- 各方式の長所を組み合わせ
- 状況に応じて最適な方式を選択
- 実装複雑度は高い

### 8.4 案D: Zero-Knowledge Discovery (研究レベル)

**コンセプト**: 「探しています」と言わずに「見つかりました」だけを伝える

```
PIR (Private Information Retrieval) を応用
→ ネットワーク全体が「データベース」
→ 何を取得したかをDB側に知らせずに情報取得
```

**評価**:
- C2 ✅✅, 計算コスト ❌❌

## 13. ブレイクスルー候補案

### 13.1 案E: Broadcast Veil（全員が受信者）

**コンセプト**: 「宛先」という概念を消す

```
1. メッセージを暗号化（共有秘密で）
2. 「匿名パケット」としてネットワークに放流（宛先なし、識別子なし）
3. 全ノードがGossipで受信
4. 各ノードが「自分宛てか？」を判定
5. 復号できた人だけがメッセージを読める
```

**受信判定の効率化**:
```
Tag = HMAC(SharedSecret, Nonce)[0:4]  // 4バイトのブラインドタグ

受信者は全コンタクトのTagを計算
→ 一致したら復号を試す
→ 誤検知率 1/2^32、実用上問題なし
```

**評価**:
| 特性 | 評価 |
|:---|:---|
| 検索問題 | ✅ 完全解決（検索が存在しない） |
| 受信者特定 | ✅ 完全解決（全員が受信者） |
| 帯域コスト | ⚠️ 高い（全ノードに拡散） |
| スケール | ⚠️ 参加者数に応じて帯域増加 |

---

### 13.2 案F: シュレーディンガーのMailbox

**コンセプト**: 「受信者が取得するまで、誰宛てか確定しない」

```
従来:
  Mailbox Key = Hash(ReceiverID)
  → Mailboxは「誰宛てか」を知っている

新方式:
  Mailbox Key = Hash(RandomNonce)
  → 完全にランダム、誰宛てか分からない

受信者の発見方法:
  1. 送信者が Nonce を暗号化
  2. 暗号化 Nonce を Gossip で配信（Hint）
  3. 受信者は Hint を復号し、Nonce を得てMailboxを特定
```

**データフロー**:
```
Alice → Bob にメッセージを送る場合:

1. Alice: Nonce = random()
2. Alice: Mailbox_Key = Hash(Nonce)
3. Alice: メッセージを Mailbox[Mailbox_Key] に PUT
4. Alice: Hint = Enc(SharedSecret, Nonce || MessageID)
5. Alice: Hint を Gossip でブロードキャスト

6. Bob: Gossip から復号できる Hint を探す
7. Bob: Nonce を得る → Mailbox_Key を計算
8. Bob: Mailbox[Mailbox_Key] から GET

結果:
  - Mailbox は「ランダムなキーへのアクセス」としか見えない
  - Gossip も「ランダムなバイト列」としか見えない
  - 誰も Alice → Bob の関係を知らない
```

**評価**:
| 特性 | 評価 |
|:---|:---|
| 検索問題 | ✅ 完全解決 |
| 受信者特定 | ✅ 完全解決 |
| 帯域コスト | ○ Hint配信のみGossip |
| 遅延 | ◎ 低遅延維持可能 |

## 14. DHT-Free アーキテクチャ

### 14.1 問題: DHTでIPが露出する

```
従来のKademlia DHT:

1. 参加時:
   「俺は NodeID=0xABC で、IP=203.0.113.55:8080 だ」と宣言

2. ルーティングテーブル:
   全ノードが「誰がどのIPか」のリストを持つ

3. 検索時:
   FIND_NODE / FIND_VALUE で他ノードに問い合わせ
   → 問い合わせ元のIPが分かる

結果:
   DHTに参加した時点で「あなたのIPは〇〇」と世界に公開
```

### 14.2 解決: Public Relay Network

**設計思想**: ユーザーは中継ノード経由でのみ通信

```
┌─────────────────────────────────────────────────────────────┐
│              AETHER PUBLIC RELAY NETWORK                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [ユーザー層]                                               │
│      │                                                      │
│      │ IP見える（入口Relayのみ）                           │
│      ▼                                                      │
│  [入口 Relay]  ←─── ハードコードまたは紹介で取得           │
│      │                                                      │
│      │ Onion暗号化                                          │
│      ▼                                                      │
│  [中間 Relay]                                               │
│      │                                                      │
│      │ Onion暗号化                                          │
│      ▼                                                      │
│  [出口 Relay]                                               │
│      │                                                      │
│      │ 宛先へ転送（E2EE維持）                              │
│      ▼                                                      │
│  [Mailbox / 相手のRelay]                                    │
│                                                             │
│  各Relayが知っていること:                                   │
│    入口: ユーザーのIP、次のRelay                           │
│    中間: 前のRelay、次のRelay                               │
│    出口: 前のRelay、宛先                                    │
│                                                             │
│  → 誰も全体像を知らない                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 14.3 Gossip + Relay の組み合わせ（最終形）

```
┌─────────────────────────────────────────────────────────────┐
│           RELAY-BACKED GOSSIP                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. ユーザーは Relay Network に Onion 接続                 │
│     → IP は入口 Relay にしか見えない                       │
│                                                             │
│  2. メッセージは Gossip で Relay Network 内に拡散           │
│     → 「誰が送ったか」追跡不能                             │
│                                                             │
│  3. 受信者も Relay 経由で Gossip を受信                     │
│     → 「誰が受け取ったか」追跡不能                         │
│                                                             │
│  4. メッセージは暗号化 + ブラインドタグ                     │
│     → 復号できるのは受信者だけ                             │
│                                                             │
│  達成される特性:                                            │
│    ✅ ID交換だけで通信可能                                  │
│    ✅ 直接会う必要なし                                      │
│    ✅ 通信相手にIPバレない                                  │
│    ✅ Relay にも「誰と誰」分からない                       │
│    ✅ 完全分散（中央サーバーなし）                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 15. 遅延分析

### 15.1 3-Hop Onion の遅延見積もり

```
各Hop の遅延（片道）:
  - 同一国内: 10-30ms
  - 近い大陸間: 50-80ms
  - 遠い大陸間: 100-150ms

3-Hop 片道:
  最良（国内）: 3 × 20ms = 60ms
  平均:         3 × 40ms = 120ms
  最悪（世界中）: 3 × 100ms = 300ms
```

### 15.2 パターン別遅延

| パターン | 構成 | 遅延 |
|:---|:---|:---|
| 直接配信 | 送信Onion(120ms) + Mailbox(10ms) + 受信Onion(120ms) | **約250ms** |
| Gossip配信 | 送信Onion(120ms) + Gossip(100ms) + 受信Onion(120ms) | **約340ms** |
| 最適化版 | 双方向Onion(240ms) | **約240ms** |

### 15.3 既存技術との比較

| 技術 | 遅延 | 備考 |
|:---|:---|:---|
| Telegram | 50-100ms | 中央サーバー経由 |
| Signal | 100-200ms | 中央サーバー経由 |
| Tor Hidden Service | 500-2000ms | 6-Hop (3+3) |
| I2P | 500-3000ms | Garlic Routing |
| **AETHER (提案)** | **200-350ms** | 3-Hop Onion |

### 15.4 結論

```
目標: 200-500ms

結果:
  最良ケース: 150ms
  平均ケース: 250-300ms ✅
  最悪ケース: 400-500ms

→ 許容範囲内で達成可能
→ Torの半分以下の遅延
```
