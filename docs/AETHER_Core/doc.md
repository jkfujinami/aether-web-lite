**次世代完全分散型・隠蔽通信プロトコル：Project AETHER**

**〜 メタデータ完全秘匿と耐量子・対検閲アーキテクチャ 〜**

## **1\. エグゼクティブ・サマリー**

Project AETHERは、既存のメッセージングアプリ（Telegram, Signal等）が抱える「中央サーバーへの依存」および「トラフィック解析によるメタデータの露出」という致命的な脆弱性を解決する。Rust言語を用いて、高スループットなQUICプロトコルを基盤とした完全分散型P2Pネットワークを構築し、国家レベルのディープ・パケット・インスペクション（DPI）を回避しつつ、ミリ秒単位の高速通信を実現する。

## ---

**2\. 技術スタック・アーキテクチャ**

### **2.1 トランスポート層：Stealth-QUIC & WebRTC Camouflage**

検閲耐性を極限まで高めるため、現代のビデオ会議プロトコルへの完全な偽装を行う。

* **WebRTC擬態**: パケットのサイズ分布、送信タイミング、ハンドシェイク手順を **Zoom/Google Meetの暗号化通信 (UDP/SRTP)** と統計的に区別不能にする。YouTube（ダウンストリーム主体）への偽装から、双方向性が自然なWebRTCへの偽装へ戦略を転換。
* **パケットパディング**: コンテンツサイズを隠蔽するため、通話の静寂やノイズに見せかけたダミーデータを動的に挿入。

### **2.2 ネットワーク層：Hybrid Onion-Gossip & Mailbox**

中央サーバー不在による「オフライン時の不達」と「メタデータ露出」を解決する分散アーキテクチャ。

* **Blind Mailbox Swarm (オフライン配送)**:
    * 受信者がオフラインの場合、暗号化されたメッセージを**「Blind Mailbox（匿名私書箱）」**として機能する5〜7つのランダムな高稼働ノードに分散保管させる。
    * **Storage Model (Ring Buffer / FIFO)**:
        * Mailboxは受信者ごとに固定容量（例: 50MB）のリングバッファを割り当てる。
        * 容量を超過した場合、**最も古い断片から順に強制削除**され、新しいメッセージのためにスペースを確保する。これによりストレージ枯渇攻撃（DoS）を防ぐ。
    * **Payload Agnostic**:
        * テキスト、画像、Zipファイルなどの形式を問わず、全てのデータは暗号化された不透明なバイナリBlobとして扱われる。Mailbox側は内容を一切関知しない。
    * **Erasure Coding (消失訂正符号)**:
        * Blobを「データ断片」と「パリティ断片」に分割（例: 3つのデータ + 2つのパリティ）して各ノードに配布。**任意の3つのノード**からデータを回収できれば、残りがダウンしても完全な復元が可能。
    * **Burn-on-Read (Atomic Deletion)**:
        * 受信者がデータをダウンロードし、復号・検証に成功した後、署名付きの「削除命令（Receipt）」を送信する。Mailboxはこの命令を受信次第、即座に当該データを物理削除する。
* **3-Hop Onion Routing**: 送信元・Mailbox・受信者の間の通信はすべて多層暗号化され、IPアドレスは隠蔽される。

### **2.3 接続・電力管理：Adaptive Ghost Mode**

モバイル環境における「バッテリー消費」と「常時接続」のトレードオフを解消する。

* **Adaptive Ghost Mode (モバイル向け)**:
    * **Active**: アプリ起動時はリレーノードとして機能し、Cover Trafficを生成。
    * **Ghost**: バックグラウンド時は**消費電力を最小化**。定期的（数分〜数十分おき）にMailboxを一瞬だけポーリングし、着信の有無を確認する「準リアルタイム」動作へ移行。OSのPush通知サービス（FCM/APNs）には依存しない。

**3\. セキュリティ・匿名性定義**

### **3.1 耐量子エンドツーエンド暗号化 (PQ-E2EE)**

* **鍵交換**: Kyber-768 (ML-KEM) \+ X25519 のハイブリッド方式。量子コンピュータによる遡及的解読を防御。
* **二重ラチェット (Double Ratchet)**:
    * Signalプロトコルを拡張し、メッセージ毎の鍵更新（Symmetric Ratchet）と、返信時のルート鍵更新（DH Ratchet）を組み合わせる。
    * **Resilience (再送制御)**: メッセージロストや順序逆転が発生しても、チェーン計算により復号を継続。ACK/NACKシグナルを用いた再送ロジックをプロトコルレベルで統合し、完全性を担保する。
* **自己主権型ID (SSI)**: ユーザー識別子はEd25519公開鍵のハッシュのみ。電話番号、メールアドレス、中央サーバーへの登録は一切不要。

### **3.2 メタデータ・シュレッディング**

* **Cover Traffic (PC/Active時)**: 通信の有無に関わらず、微弱なダミーパケットを送信し続けることで、「いつ、誰が、誰と通信を開始したか」というタイミング解析を無効化。
* **ジッター制御**: パケット送信間隔にミリ秒単位の物理的な揺らぎを与え、機械学習によるフロー識別を回避。

**4. 脅威モデルと検証 (Threat Model & Verification)**

### **4.1 想定される敵対者 (Adversary Landscape)**

AETHERは以下の能力を持つ敵対者を想定し、防御策を講じる。

| レベル | 敵対者 (Adversary) | 想定される能力 (Capabilities) | AETHERの対策 |
| :--- | :--- | :--- | :--- |
| **L1** | **Local Observer**<br>(カフェWi-Fi, 情シス) | パケット盗聴、DNS監視 | 全通信のWebRTC擬態化、DNSレス(DHT) |
| **L2** | **ISP / Carrier**<br>(通信事業者) | メタデータ収集(DPI)、契約者紐付け | 3-Hop Onion Routing、Cover Traffic |
| **L3** | **Global Passive Adversary**<br>(国家諜報機関, AS監視) | インターネット全域のトラフィック相関分析 | **非同期Mailboxによる時間的分断** (Temporal Decoupling) |
| **L4** | **Active Attacker**<br>(国家ハッカー, 検閲機関) | 悪質ノード大量配備(Sybil)、端末押収 | Node Age評価、**Duress PIN / Panic PIN** |

### **4.2 匿名性メカニズムの検証**

1.  **相関攻撃 (Correlation Attack) への耐性:**
    *   **課題:** Tor等の低遅延ネットワークは、入口と出口のパケットタイミングからユーザーを特定されるリスクがある。
    *   **解決策:** AETHERはMailboxを用いた**非同期通信**を前提とする。送信と受信の間に意図的な遅延（数分〜数時間）と、受信タイミングの任意性（Pull型）を挟むことで、時間軸上の相関を数学的に断絶する。

2.  **プロトコル識別 (DPI) への耐性:**
    *   **課題:** 単純なUDP通信は、「Zoomらしくない（ビットレートが一定すぎる）」として機械学習で遮断される。
    *   **解決策:** **Model-Based Traffic Shaping**を実装。VBR（可変ビットレート）ビデオコーデックの統計モデル（変動パターン）に基づき、ダミーパケットを動的に注入してトラフィック形状を整形する。

### **4.3 ローカルセキュリティと否認可能性 (Local Security & Plausible Deniability)**

端末押収（Forensics）および強要（Duress）に対抗するための多層防御。
**「OS標準のハードウェアセキュリティ(Secure Enclave/TPM)がある場合」**と**「ない場合」**で挙動を分岐する。

*   **Duress PIN / Panic PIN (ハードウェア支援推奨)**:
    *   ログイン画面のPINコードによって、復号される「世界」を分岐させる。
    *   **通常PIN**: 正規のデータ領域を復号し、アプリを起動。
    *   **Decoy PIN (囮)**: 空の、あるいは無害なダミーデータ領域を復号。「何も隠していない」ように見せかける。
    *   **Panic PIN (緊急)**: **暗号化解除キー（Master Key）を即座に物理削除**し、アプリをクラッシュさせる。以後、データは二度と復元不能になる（電子のゴミ化）。
*   **環境依存のポリシー**:
    *   TPM/Secure Enclave非搭載機では、GPU解析を防ぐために**「8文字以上の強力なパスワード」**を強制する（4桁PINは禁止）。

**5\. 実装ロードマップ (Rust実装計画)**

1. **Phase 1: Foundation (1-2ヶ月)**
   * `libp2p` + `quinn` (QUIC) をベースとしたP2Pスタックの構築。
   * **Blind Mailbox** のプロトタイプ実装（オフラインメッセージングの確立）。
2. **Phase 2: Stealth & Security (2-3ヶ月)**
   * `uTLS` 統合によるTLS指紋偽装と、WebRTCパケットの模倣ロジック実装。
   * Kyberライブラリによる耐量子暗号化層の実装。
3. **Phase 3: Routing & Optimization (2-3ヶ月)**
   * 3ホップ・オニオンルーティングの実装。
   * モバイル向け **Ghost Mode** (省電力ポーリング) の実装と調整。
4. **Phase 4: Alpha Release (1-2ヶ月)**
   * ポート開放不要環境（NAT越え）での大規模接続テスト。
   * 検閲環境を模したラボでのDPI回避性能検証。

**6\. 結論**

AETHERは、現代のインターネット監視社会に対する\*\*「技術的主権（Technical Sovereignty）」\*\*の確立を目指す。Rustのメモリ安全性と低レイヤー制御能力を最大限に活用し、速度、匿名性、そして利便性の三要素を妥協なく統合した。これは、単なるアプリではなく、自由な通信を支える「不可視のインフラ」である。
