# Fractal Hierarchical Network (FHN)

## 概要

AETHERにおけるノード探索およびネットワーク維持プロトコル。
自己相似（フラクタル）な階層構造と、グループ内での継承順位（Ranking）に基づく自律的な管理により、高い匿名性、スケーラビリティ、耐障害性を実現する。

## 構造

### フラクタル階層構造

ネットワークは同一のルールで管理される「グループ（Cell）」の階層によって構成される。

- **Level 0 (末端)**: 一般ユーザーによって構成される最小単位のグループ。
- **Level 1**: Level 0 の各グループから選出されたリーダーによって構成されるグループ。
- **Level N**: Level N-1 のリーダーによって構成される上位グループ。

```
[Level 2 Group] (Top Layer)
   メンバー: { Leader_A, Leader_B, Leader_C ... }
      │
      │ Leader_A は Group A の代表
      ▼
[Level 1 Group A]
   メンバー: { Leader_a1, Leader_a2, Leader_a3 ... }
      │
      │ Leader_a1 は Group a1 の代表
      ▼
[Level 0 Group a1] (Bottom Layer)
   メンバー: { User_1, User_2, User_3 ... }
```

### グループ構成ルール

すべての階層において、グループは以下の統一ルールに従う。

1. **定員**: 20〜30ノード
   - 合意形成の速度と、ビザンチン耐性（悪意あるノードへの耐性）のバランスが最適な規模。
   - 20人未満になれば統合(Merge)、30人を超えれば分割(Split)する。

2. **地理的分散**:
   - 可能な限り異なる国・帯域のノードを混在させる（国単位の検閲耐性）。

## リーダー選出と継承 (Succession)

選挙や投票による遅延を防ぐため、常に「継承順位」を共有しておく。

### ランキングシステム

グループ内の全メンバーは、以下の指標に基づきスコア付けされ、順位決定される。
- **Uptime**: 稼働時間（長いほど信頼できる）
- **Bandwidth**:帯域幅
- **Reputation**: 過去の振る舞い（正しく中継したか等）

### 役割

| 順位 | 役割 | 責務 |
|------|------|------|
| **Rank 1** | **Leader** | ・上位階層のグループに参加する<br>・自グループと上位層のルーティングを行う<br>・メンバー管理（参加・離脱） |
| **Rank 2** | **Vice Leader** | ・Leader の死活監視<br>・Leader ダウン時に即座に昇格 |
| **Rank 3~** | **Member** | ・自グループ内の情報の保持<br>・下位層（自分が代表しているグループ）の管理 |

### 障害時の自動継承

Leader (Rank 1) がオフラインになった場合：
1. Rank 2 がタイムアウトを検知。
2. Rank 2 が **即座に Leader に昇格** し、メンバーに通知。
3. Rank 3 が Vice Leader に昇格。
4. 旧 Leader はリストから除外され、再接続時は最下位からスタート。

**→ "選挙" が不要なため、ダウンタイムがほぼゼロ。**

## 可視性 (Visibility) の隔離

「誰が何を知っているか」を厳格に制限し、全貌の把握を困難にする。

### 1. Leader (Rank 1) の視界
- **上位**: 上位グループのメンバーが見える。
- **自グループ**: 自グループの全員が見える。
- **下位**: *見えない*（自分が元いた下位グループは見えているが、役割としては上位への接続が主）。

### 2. 一般メンバー (Leader以外) の視界
- **上位**: *見えない*（Leader しかアクセスできない）。
- **自グループ**: 自グループの全員が見える。
- **下位**: **見える**。
  - メンバーは「下位グループの代表」として参加しているため、自分が管理する下位グループの全メンバー情報を保持している。

### この設計の意図
「メンバーは下が見える」ことにより、検索クエリが上から降りてきた際、
**「その宛先なら、私の部下（下位グループ）にいるよ」** と即答できる。

## ルーティングと探索 (Discovery)

### 探索フロー (Query Routing)

Alice (Level 0, Group A) が Bob (Level 0, Group B) を探す場合：

1. **上昇フェーズ (Ascent)**
   - Alice は自グループの Leader にクエリを投げる。
   - Leader は自グループに Bob がいなければ、上位グループへ投げる。
   - これを繰り返し、Bob の所属ツリーを知る共通の祖先（あるいは最上位）まで登る。

2. **下降フェーズ (Descent)**
   - 共通の祖先グループで、「Bob のID範囲」を担当しているメンバー（Bobの所属するツリーの代表者）が見つかる。
   - そのメンバーは「自分の下位グループにいる」と分かるため、クエリを下ろす。
   - これを繰り返し、Bob のいる Level 0 グループに到達する。

3. **応答 (Response)**
   - Bob の TunnelEndpoint が発見される。
   - 結果は来た経路を逆にたどるか、あるいは直接（匿名トンネル経由で）Alice に返される。

### クエリの匿名化
全てのクエリとレスポンスは **Onion Routing / Garlic Routing** でカプセル化される。
中継する各リーダーは、「誰を探しているか」は分かるが、「誰が探しているか（Aliceの実IP）」は分からない。

## セキュリティ強度

| 脅威 | 耐性 | 理由 |
|------|------|------|
| **DHT汚染** | 極めて高い | 固定的なIDではなく「スコア」によるRanking制のため、攻撃者が即座にリーダーになれない（Uptimeを稼ぐ必要がある）。 |
| **全IP収集** | 不可能 | 自分の所属グループと直下グループしか見えない。全体を見渡せる "神の視点" を持つノードが存在しない。 |
| **リーダー攻撃** | 高い | Leaderを落としても、即座にRank 2が昇格するためネットワークは止まらない。 |
| **Sybil攻撃** | 高い | 1つのグループに入れる人数が制限されており、かつ多国籍に分散されるため、多数派工作が困難。 |

## グループ分割アルゴリズム

### 分割トリガー

- グループメンバーが **31人** を超えた時点で分割を実行

### 決定論的分割

誰も操作できない、検証可能な分割を実現する。

```
分割手順:

1. シード生成:
   seed = SHA256(
       グループID +
       分割時点のタイムスタンプ +
       全メンバーの公開鍵をソートして連結
   )

2. シードで乱数生成器 (ChaCha20) を初期化

3. メンバーリストをシャッフル

4. 前半 → 新グループ A
   後半 → 新グループ B

5. 各グループでリーダーを再選出（スコア順）
```

### なぜ操作不可能か

```
シードの構成要素:
├── 全メンバーの公開鍵 ← 1人でも変わると結果が変わる
├── タイムスタンプ ← 分割タイミングは予測困難
└── グループID ← 固定値

攻撃者が操作するには:
├── 特定のメンバー構成を作る → 他者の参加/離脱はコントロール不可
├── 特定のタイミングを狙う → 誰かが先に31人目になるかも
└── 事実上不可能
```

### 検証可能性

```
任意のノードが分割結果を検証可能:

1. 分割時点のメンバーリストを取得
2. 同じシード計算を実行
3. 同じ結果が得られることを確認

→ リーダーが不正な分割をしても即座にバレる
```

## 段階的地理分散要件

ネットワークの成長段階に応じて、地理分散要件を調整する。

### フェーズ定義

| フェーズ | ネットワーク規模 | 地理分散要件 | 理由 |
|----------|-----------------|-------------|------|
| **Phase 1** | 100人未満 | なし | まず動かすことを優先 |
| **Phase 2** | 100-1000人 | 推奨 | 可能なら分散、無理でも許容 |
| **Phase 3** | 1000人以上 | 必須 | セキュリティ重視 |

### Phase 1: 初期段階

```
初期状態の現実:
├── 日本人 50人
├── アメリカ 5人
├── その他 2人

→ 地理分散を強制すると動かない
→ 要件を免除
```

### Phase 2: 成長段階

```
地理分散チェック:
├── 理想: グループ内に 3地域以上
├── 許容: 2地域でも OK
├── 警告: 1地域のみ（改善を促す）

上位ノード選出:
├── 可能なら異なる地域から
├── 無理なら同一地域でも許容
```

### Phase 3: 成熟段階

```
地理分散要件:
├── グループ内に最低 3地域
├── 同一地域が過半数を占めてはならない

上位ノード (Level 2+):
├── 異なる地域から選出
├── 同一地域からの上位ノードは 2人まで
```

### フェーズ判定

```
現在のフェーズ = ネットワーク全体のノード数に基づく

判定方法:
├── 上位リーダーが集計した推定値を使用
├── または Bootstrap ノードが把握している接続数
```

## Bootstrap リストの信頼性と配布戦略

初回接続先（Bootstrapノード）を安全かつ検閲耐性を持って配布するための多層防御策。

### 1. マルチソース・シード (Multi-Source Seeding)

単一のサーバーに依存せず、複数の独立したチャネルからシードリストを取得し、統合する。

| ソース種別 | 方法 | 特徴 |
|------------|------|------|
| **GitHub/GitLab** | 特定リポジトリ内の署名付きファイル | 検閲困難、高可用性 |
| **DNS TXT** | `seed.aether-net.org` 等のTXTレコード | 取得が高速、ブロッキング容易 |
| **IPFS** | 特定のIPNSアドレス | 永続的、分散型 |
| **HTTPS API** | ボランティア運営のエンドポイント | 手軽だがブロックされやすい |

#### 取得フロー
1. アプリ起動時、上記ソースに対し並行リクエストを送信。
2. 取得できたリストの署名を検証（後述）。
3. 有効なリストを統合（重複排除）し、応答が速い順に接続を試行。

### 2. 開発者署名 (Developer Signature)

配布されるリストが汚染されていないことを保証する。

- **署名プロセス**:
  - シードリストは、開発チームの秘密鍵で署名される。
  - アプリ内には開発チームの公開鍵がハードコードされている。
- **検証プロセス**:
  - どのソースから取得した場合でも、署名検証に成功しない限りそのリストは破棄される。
  - これにより、DNSハイジャックやGitHub侵害時でも、偽のIPリストを掴まされるリスクを排除する。

### 3. Bridge ノード (秘密の入り口)

国家レベルの検閲等で、公開されているBootstrapリストが全て遮断された場合の最後の手段。

- **非公開配布**: IPアドレスは公開リストには載せない。メール、SNS、信頼できる友人などを通じて個別に共有される。
- **カモフラージュ**: Bridgeへの接続は、通常のHTTPSやSSHトラフィックに見えるよう難読化（Obsfucation）される。
- **ユーザーによる追加**: ユーザーは手動で「入手したBridgeアドレス」をアプリに入力できる。

### 4. 信頼の確立 (Trust on First Use - TOFU)

初回接続に成功した後、アプリはネットワークから最新の「信頼できるノードリスト」を取得し、ローカルにキャッシュする。
2回目以降の起動では、外部ソース（GitHub等）に依存せず、このローカルキャッシュを使用してP2Pネットワークに再参加する。

## 階層別役割分担

### 設計思想

「公開度」と「リスク」を分離する。

| 階層 | 公開度 | 役割の危険度 | 担当業務 |
|------|--------|-------------|---------|
| **上位 (Level 2+)** | 高い | 低い | Bootstrap、ルーティング案内 |
| **下位 (Level 0-1)** | 低い | 高い | 実際の通信中継、Tunnel処理 |

### 上位ノードの責務

```
上位ノード (Level 2, 3, ...):

  [Bootstrap 機能]
  ├── 新規ノードの最初の接続先として公開
  ├── 適切な Level 0 グループを紹介
  └── 紹介後は新規ノードの活動を追跡しない

  [ルーティング機能]
  ├── クエリを正しい下位ツリーへ振り分け
  ├── 暗号化されたクエリの中身は見えない
  └── 「どのツリーに属するか」だけを判断

→ 公開されてもリスクが低い
→ 攻撃されても「案内役」が止まるだけで通信は継続可能
```

### 下位ノードの責務

```
下位ノード (Level 0, 1):

  [通信中継]
  ├── Inbound/Outbound Tunnel の中継点
  ├── 暗号化層の追加・削除
  └── 実際のメッセージ転送

  [リスクの性質]
  ├── 通信内容に関わる処理を担当
  ├── 法的・技術的な攻撃の対象になり得る
  └── しかし「誰がやっているか」は外部から見えない

→ 危険な役割だが、匿名性で保護される
```

## クールダウン機構

### 目的

- 上位ノードの長期居座りを防止
- 権力の固定化・共謀リスクを低減
- 役割の公平な分担

### ライフサイクル

```
ノードのライフサイクル:

[参加] → Level 0 (最下位からスタート)
              ↓ スコア上昇
         Level 1 リーダーに昇格
              ↓ さらにスコア上昇
         Level 2 リーダーに昇格
              ↓
         ┌────────────────────────┐
         │  上位在籍期間: 最大7日   │
         └────────────────────────┘
              ↓ 期間満了
         クールダウン発動
              ↓
         Level 0 に強制降格
         (スコアは一定ペナルティ付きでリセット)
              ↓
         クールダウン期間: 3日
         (この間は昇格不可)
              ↓
         通常運用に復帰、再び昇格可能
```

### パラメータ案

| パラメータ | 値 | 理由 |
|-----------|-----|------|
| 上位最大在籍期間 | 7日 | 攻撃者が長期掌握する時間を制限 |
| クールダウン期間 | 3日 | 連続昇格による固定化を防止 |
| 降格時スコアリセット | 50% | 完全リセットだと古参が不利すぎる |

## Bootstrap フロー

### 新規ノード参加手順

```
新規ノード X の参加:

1. X が公開 Bootstrap エンドポイントに接続
   ├── Bootstrap エンドポイント = 上位ノード (Level 2+) の一部
   └── IPは公開されているが、安全な役割のみ

2. Bootstrap が適切な Level 0 グループを選択
   ├── 地理的分散を考慮（Xと同じ国に偏らない）
   ├── グループの現在人数を考慮（満員なら別へ）
   └── 負荷分散を考慮

3. Bootstrap が Level 0 リーダーの接続情報を X に返却
   ├── リーダーの TunnelEndpoint (IPは隠蔽可能)
   └── グループの公開鍵

4. X が Level 0 リーダーに参加申請
   ├── 自己紹介（公開鍵、希望する国籍情報など）
   └── PoW (Proof of Work) で Sybil 対策

5. リーダーが承認し、グループメンバーリストを共有

6. X は Level 0 の一般メンバーとして活動開始


→ Bootstrap は「どの Level 0 に振ったか」だけ知っている
→ X のその後の通信内容や活動は一切追跡できない
```

### Bootstrap ノードの選出

```
どのノードが Bootstrap 役を担うか:

条件:
  - Level 2 以上のノード
  - 高い Uptime スコア
  - 安定した帯域

ローテーション:
  - 複数の Bootstrap ノードが並行稼働
  - 1日ごとにローテーション
  - 特定ノードへの負荷集中を防止
```

## スコアシステム

### スコアの構成要素

| 要素 | 意味 | 重み(案) |
|------|------|---------|
| **Uptime** | 稼働時間 | 20% |
| **Bandwidth** | 中継したデータ量 | 15% |
| **Latency** | 応答速度 | 10% |
| **Reliability** | 中継成功率 | 20% |
| **Age** | ネットワーク参加期間 | 10% |
| **Diversity** | 地理的貢献 | 10% |
| **RegionalStability** | 地域の安定度 | 15% |

### 地域安定度 (RegionalStability) の定量化

ノードの所在地域に基づき、政治的・法的安定度をスコア化する。

#### 地域ティア分類

| ティア | スコア係数 | 基準 | 例 |
|--------|-----------|------|-----|
| **Tier 1** | 1.0 | インターネット自由度が高い、法的保護が強い | スイス、アイスランド、オランダ |
| **Tier 2** | 0.8 | 比較的安定、一部規制あり | 日本、ドイツ、カナダ |
| **Tier 3** | 0.6 | 規制が増加傾向、監視リスクあり | アメリカ、イギリス、オーストラリア |
| **Tier 4** | 0.4 | 厳しい規制、VPN規制など | ロシア、トルコ、UAE |
| **Tier 5** | 0.2 | 強権的検閲、匿名通信が違法 | 中国、北朝鮮、イラン |

#### 評価指標

以下の公開指標を参考にスコアを算出:

1. **Freedom House - Freedom on the Net**
   - インターネット自由度の年次レポート
   - 0-100 のスコア

2. **Reporters Without Borders - Press Freedom Index**
   - 報道の自由度

3. **VPN合法性**
   - 合法 / グレー / 違法

4. **データローカライゼーション法**
   - なし / あり / 厳格

#### スコア計算例

```
ノード B (日本):
├── Tier 2 → 係数 0.8
├── RegionalStability = 0.8 × 100 = 80

ノード C (スイス):
├── Tier 1 → 係数 1.0
├── RegionalStability = 1.0 × 100 = 100

ノード D (中国):
├── Tier 5 → 係数 0.2
├── RegionalStability = 0.2 × 100 = 20
```

#### 上位ノード選出への影響

```
上位ノード (Level 2+) の条件:

必須:
├── RegionalStability >= 60 (Tier 3 以上)
└── Tier 4, 5 のノードは上位になれない

理由:
├── 強権国家のノードがBootstrapになると危険
├── 法的に押収・監視されるリスク
└── 上位は「公開される」ので安定地域から選出
```

### スコア証明方式: 署名付き証拠の束

ブロックチェーンではなく、各ノードが「署名付き証拠」を個人で保管する方式を採用。

```
ノード B の証拠ファイル:
├── Heartbeat証明 × 1000件 (過去1ヶ月分)
├── 中継受領証 × 500件
├── 到達確認 × 300件
└── 全て第三者の署名付き
```

**利点**:
- グループに依存しない（ローテーションしても持ち歩ける）
- 自分だけが保管すればよい（全体共有不要）
- 必要な時だけ検証（常時同期不要）

### 各スコアの証明方法

#### 1. Uptime (稼働時間)

```
[Heartbeat Challenge-Response]

A → B: Challenge (ランダム値 + タイムスタンプ + A の署名)
B → A: Response (チャレンジ + B の署名)

B が保管する証拠:
[Heartbeat Proof]
├── timestamp: 2026-01-29 12:00:00
├── challenger: A の公開鍵
├── challenge: ランダム値
├── responder: B の公開鍵
└── signature: A の署名 (B が応答したことの証明)
```

#### 2. Bandwidth (中継量)

```
[中継受領証]

パケット中継時:
A → B → C

C が B に発行:
[Receipt]
├── from: B の公開鍵
├── to: C の公開鍵
├── size: 1024 bytes
├── timestamp: 2026-01-29 12:00:00
└── signature: C の署名
```

#### 3. Latency (応答速度)

```
[Ping-Pong 記録]

複数ノードからの測定結果を集計:
├── A からの RTT: 50ms
├── C からの RTT: 45ms
├── D からの RTT: 55ms
└── 中央値: 50ms

※ 単一ノードの申告は信用しない（共謀対策）
```

#### 4. Reliability (信頼性)

```
[到達確認]

送信者が最終的に「届いた」ことを確認:
[Delivery Confirmation]
├── tunnel_id: xxx
├── status: SUCCESS
├── relay_path: [A, B, C]
├── timestamp: 2026-01-29 12:00:00
└── signature: 送信者の署名

→ 経路上の全ノード (A, B, C) がこの証拠を保管可能
```

#### 5. Age (参加期間)

```
最古の証拠のタイムスタンプ = 参加開始日

ノード B の最古証拠: 2025-06-01
現在: 2026-01-29
→ Age = 約8ヶ月
```

#### 6. Diversity (地理的貢献)

```
中継先の地域分布:
├── 日本: 100件の受領証
├── ドイツ: 80件
├── ブラジル: 60件
└── 3地域 = Diversity スコア高
```

### 改ざん不可能性

#### なぜ改ざんできないか

```
証拠の構造:
[Proof]
├── data: 元データ
└── signature: 第三者の署名 ← ここがポイント

改ざんしようとすると:
├── data を変更 → signature が無効になる
├── signature を偽造 → 第三者の秘密鍵がないので不可能
└── 結論: 改ざん不可能
```

#### グループ移動後も有効な理由

```
証拠に含まれるもの:
├── 公開鍵 (検証可能)
├── データ
├── 署名

含まれないもの:
├── グループ情報
├── IP アドレス

→ どのグループでも署名検証だけで有効性を確認できる
```

### スコア検証フロー

```
ノード B が新グループ Y に参加する場合:

1. B が証拠の束を Y に提示
   「過去1ヶ月の Heartbeat 証明 1000件、受領証 500件...」

2. Y がサンプリング検証
   ├── ランダムに 10-20件を選択
   ├── 署名を検証
   └── 全て有効なら OK

3. Y が証拠からスコアを計算
   ├── Heartbeat 1000件 → Uptime スコア
   ├── 受領証 500件 → Bandwidth スコア
   └── ...

4. B は計算されたスコアでグループに参加
```

### 悪意あるケースへの対策

| 攻撃 | 対策 |
|------|------|
| 証拠の偽造 | 第三者の秘密鍵がないので不可能 |
| 証拠の改ざん | 署名が無効になるので検出可能 |
| 古い証拠の使い回し | タイムスタンプで有効期限を設定 |
| 共謀による偽証拠 | 複数ノードからの証拠を要求 |
