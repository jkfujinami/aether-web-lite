## 課題1: 暗号化モデル → ✅ AETHERの「シュレーディンガーMailbox」で解決

### AETHERの該当設計
`discovery_protocol_design.md` セクション13.2「シュレーディンガーのMailbox」

### 解決策
2ch（公開掲示板）では「誰でも読める」必要があるため、AETHERの「共有秘密で暗号化」はそのまま使えない。
しかし、**「メタデータの秘匿（中継役Bさんに"何の板のデータか"を悟らせない）」**には完璧に使える。

- **データ本体（レスの内容）**: 平文のまま（公開掲示板なので誰でも読める）。
- **トピック情報の秘匿**: レスデータの**宛先（どの板・どのスレッド）はハッシュ化**する。
  ```
  Mailbox_Key = SHA256("board:vip" + "thread:12345")
  ```
  中継役のBさんは `Mailbox_Key = 0xA1B2C3...` というハッシュに対してデータを保管するが、
  それが「VIP板のスレッド12345」であることは**逆算不可能（一方向ハッシュ）**。
- **閲覧者だけが知っている**: スレッドのURLに `board:vip/thread:12345` という情報が含まれているため、
  URLを知っている（＝スレッドを見ようとしてアクセスした）人だけがMailbox_Keyをクライアント側で再計算して、
  DHTから正しいMailboxに辿り着ける。

### 結論
**「内容は平文、宛先はハッシュで秘匿」**。これで「誰でも読める公開掲示板」と「中継役が何の板か分からない」を両立。

## 課題2: Mailboxのデータ引き継ぎ → ✅ AETHERの「Erasure Coding + K-Replication」で解決

### AETHERの該当設計
- `doc.md` セクション2.2「Erasure Coding（消失訂正符号）」
- `detailed_implementation.md` セクション5.2「Reed-Solomon分割/復元」
- `doc.md` 「5〜7つのランダムな高稼働ノードに分散保管」

### 解決策
Mailbox担当者が1人だけだと、その人がタブを閉じた瞬間にデータが消える（単一障害点）。
AETHERのErasure Codingをそのまま適用する。

- **K-Replication（複数担当制）**:
  スレッドの `Mailbox_Key` のハッシュ値に「最も近いノード」を1人だけでなく、
  **上位K人（例: K=5）**を全員Mailbox担当にする。5人全員が同じ過去ログのコピーを持つ。
- **Erasure Coding（断片化保管）**:
  さらに保険をかけるなら、AETHERの Reed-Solomon (3データ + 2パリティ) を適用し、
  5人のうち**任意の3人が生きていれば完全な復元が可能**にする。
- **担当者の自動引き継ぎ**:
  担当者の1人がタブを閉じると、DHTのルーティングテーブルが更新され、
  次にハッシュ値が近い新しいノードが自動的に「お前が次の担当だ」と指名される。
  既存の担当者から新担当者へデータがコピーされ、K=5を維持し続ける。

### 結論
**「5人同時にタブを閉じない限り、データは絶対に消えない」**。AETHERの設計をそのまま適用可能。

## 課題3: 画像やメディアの扱い → ❌ 新規設計が必要

### AETHERの状況
AETHERの `doc.md` には「Payload Agnostic（テキスト、画像、Zipファイルなど形式を問わず全て暗号化バイナリとして扱う）」
と書かれているが、これはネイティブアプリ（Rust）前提で帯域とストレージに余裕がある設計。
ブラウザのIndexedDBに画像を大量保管するのは現実的ではない。

### 新規アイデア

#### 案A: テキスト専用（初代2ch方式）
- 画像は一切扱わない。「テキストだけで語る掲示板」というコンセプトを貫く。
- メリット: 実装が最もシンプル。ストレージ負荷が極めて軽い。
- デメリット: 令和の掲示板としては物足りない。

#### 案B: 外部リンクのみ（Imgur/Catbox方式）
- 画像は外部アップローダー（Imgur, Catbox, IPFS Gateway等）にユーザーが自身でアップし、
  P2Pネットワーク上にはURLのテキストだけを流す。
- ブラウザのUI側で `<img>` タグとして自動展開（プレビュー表示）する。
- メリット: P2Pネットワークの帯域・ストレージを圧迫しない。
- デメリット: 外部サービスへの依存。画像が消される可能性。

#### 案C: 小サイズのみP2P内蔵（サムネイル方式）
- 200KB以下の小さい画像のみ、Base64エンコードしてJSONに埋め込む。
- 大きい画像は案Bと同じく外部リンク。
- メリット: アイコンやスクリーンショット程度なら掲示板内で完結。
- デメリット: 200KBでも1000レス分なら200MBになり、IndexedDBの圧迫が懸念。

### 推奨: 案B（外部リンク）をベースに、将来的に案Cを検討。

## 課題4: PoW難易度の合意形成 → ⚠️ 部分的にAETHERから流用

### AETHERの該当設計
- `doc.md` セクション2.2「リングバッファ」に、Mailbox側でのPoW検証の記述あり。
- `detailed_implementation.md` に `pow_difficulty` の設定項目あり。

### AETHERの限界
AETHER本体では `pow_difficulty` はグローバル設定（固定値）であり、
「スレッドの勢いに応じた動的な難易度調整」は設計されていない。

### 新規アイデア: ローカル難易度計算（Bitcoin方式の簡易版）
中央管理者なしで各クライアントが「正しい難易度」を自律計算する方式。

```
各クライアントの難易度計算ロジック:

1. 自分のIndexedDBにある「直近100件のレス」のタイムスタンプを見る
2. 100件が投稿された時間幅（秒）を計算
3. 目標: 「100件あたり300秒（=3秒に1件ペース）」
4. 実際の時間幅が目標より短い（勢いが強い）→ 難易度を上げる
5. 実際の時間幅が目標より長い（過疎）→ 難易度を下げる

受信時の検証:
- 受け取ったパケットのPoWナンスが、自分が計算した難易度を満たしているか検証
- 満たしていなければドロップ（破棄）
```

全員が同じ「直近100件」を見ているため、全クライアントがほぼ同じ難易度に収束する。
スパマーが嘘の低い難易度でパケットを投げても、他の正直なノードが弾く。

## 課題5: NAT越え（TURN）の妥協点 → ⚠️ AETHERのSTUN設計を流用 + 割り切り

### AETHERの該当設計
- `detailed_implementation.md` セクション3.3「STUN」
- 「Symmetric NAT環境では機能しない可能性あり（将来的にTURNフォールバックを検討）」と明記。

### 新規アイデア: 3段階フォールバック

```
接続試行の優先順位:

1. STUN（無料）: 大半のユーザーはこれでP2P接続が成立する。
2. WebSocket経由リレー（運営鯖）:
   STUNで繋がらなかった場合、運営のWebサーバー（トラッカー）を
   「一時的なデータ中継所」として使う。
   → TURNほど高コストではない（WebSocketは軽い）。
   → ただしデータはE2EE（Ed25519署名付き）なので、運営が中身を読むことは不可能。
3. 完全に繋がらない場合: 「あなたのネットワーク環境ではP2P接続できません」と表示。
```

### 結論
**「TURNは立てない。代わりにWebSocketフォールバック（運営鯖経由の軽量リレー）を用意する」**
というのがコストとロマンのバランスが最も良い。

## 課題6: トラッカーへのIP収集攻撃 → ⚠️ AETHERのBootstrap設計を参考に防御

### AETHERの該当設計
- `hierarchical_cell_network.md`「Bootstrapフロー」
  - 「Bootstrap は『どの Level 0 に振ったか』だけ知っている。X のその後の通信内容や活動は一切追跡できない」
- `discovery_protocol_design.md` セクション5.1 「Sybil耐性」

### 新規アイデア: トラッカーの「紹介制限」

```
攻撃シナリオ:
  攻撃者が1秒に10,000回アクセス → 全参加者のIPリストを吸い出す

防御策:
  1. レートリミット: 同一IPからの接続は30秒に1回まで。
  2. 紹介上限: 1回の接続で紹介するピアは最大6人。
     → 10,000回アクセスしても 60,000件（重複あり）で、全員のIPは得られない。
  3. 軽量PoW: トラッカーへの初回接続時にも、
     ブラウザ側で0.5秒程度の軽いPoWを要求する。
     → ボットによる大量アクセスのコストを引き上げる。
  4. トラッカー側のログ破棄: 紹介した相手のIPはメモリ上でのみ保持し、
     WebSocket切断と同時に即座に破棄する（ディスクに書かない）。
```

### 結論
完全な防御は不可能だが、**「レートリミット + 軽量PoW + ログ即時破棄」**の3層防御で、
攻撃コストを実用上問題ないレベルまで引き上げられる。
