# AETHER Web-Lite: 完全分散型ブラウザ掲示板 アーキテクチャ設計

**最終更新**: 2026-03-20
**プロジェクト**: 令和版2ch（ブラウザ完結型P2Pネットワーク）

---

## 1. エグゼクティブ・サマリー

「AETHER Web-Lite」は、ネイティブアプリ（aether-core）のような重厚な3-Hopオニオンルーティングを行わず、**Webブラウザ（WebRTC + WASM + IndexedDB）のみで完結**する完全分散型の匿名掲示板システムである。

旧来の「板・スレッド」という中央集権的サーバー概念を解体し、「役割の強制（ノードの囲い込み）」を排除した**フラットで流動的な単一P2Pメッシュ（Global Mesh）**を構築する。これにより、気軽な掲示板としてのユーザビリティを保ちながら、「どのIPがどのスレッドに書き込んだか・閲覧しているか」を数学的・構造的に秘匿する。

## 2. コア・コンセプトの再定義 (2chのパラダイムシフト)

| 旧来の概念 | 新アーキテクチャでの再定義 |
| :--- | :--- |
| **中央サーバー** | **存在しない**。シグナリング（初期接続の待ち合わせIP交換）のみ使い捨てのTracker/Nostr Relay等を利用。 |
| **板 (Board)** | トピック（`board:vip`など）に基づく**GossipSubのパブサブ・チャンネル**。 |
| **スレッド (Thread)** | 固有のトピックIDを持つ独立した**暗号化データ（JSON Blob）のストリーム**。 |
| **過去ログ (DAT)** | ブラウザの `IndexedDB` に保存されるリングバッファ構造。 |
| **保守 (Hoshu)** | 無意味な書き込みをすることではなく、**「ブラウザのタブを開きっぱなしにしてデータを抱える（Mailboxになる）」**こと。 |
| **DAT落ち** | スレッドの全参加者がタブを閉じ、**ネットワーク上からデータが物理的に完全消滅**すること。 |
| **専ブラ** | このWebサイト（HTML/JS）そのものが専ブラ兼一時サーバーとして機能する。 |

## 3. ネットワーク・トポロジーとIP秘匿性

### 3.1 Global Sparse Mesh (単一巨大メッシュ網)
サイト (`reiwa-2ch.net` 等) にアクセスした全ユーザーは、閲覧している板やスレッドに関わらず、**ランダムな少数（3〜5個）の他ブラウザ（Peer）とだけWebRTC(Data Channel)を確立**し、一つの巨大な「疎結合ネットワーク」を形成する。

### 3.2 Plausible Deniability (否認可能性) によるIP隠蔽
IPアドレスは、直接接続している3〜5人の隣接ノードには物理的に露出するが、**「誰がどのスレッドに参加しているか」は完全に秘匿される**。

- **ブラインド・ルーティング**:
  Aさんが「VIP板」に書き込むパケットは暗号化され、Aさんが繋がっている隣のノード（例: アニメ板を見ているBさん）へパスされる。Bさんは中身を知らずにさらに隣へゴシップ中継する。
- **カモフラージュ効果**:
  全てのパケットは誰かが中継（リレー）したものであるため、特定のパケットを出力したIPを見つけても、「その人が発信源（作者）」なのか「他人のパケットを中継しただけ」なのか区別できない。

### 3.3 Oblivious Relay (1-Hop 差し込みによるIP分離)
同じ板（トピック）を閲覧しているユーザー同士が直接WebRTCで繋がると、「あいつのIPは○○板の住人だ」と特定される弱点（GossipSubのデフォルト挙動の限界）がある。
これを防ぐため、AETHERの「Blind Mailbox」の概念を応用して**「無関係な第三者」を中継ハブとして間に挟み込む**。

- **Kademlia DHTによる無作為な選出（ハブの決定）**:
  「VIP板」を見たいAさんとCさんは直接繋がらない。VIP板のトピック（ハッシュ値）に一番近いNode IDを持つ無関係なBさん（例: アニメ板の閲覧者）が、システムから強制的にVIP板のMailbox（保管・中継役）に自動指名される。
- **IPと興味の分離**:
  AさんもCさんも、中継役のBさんにデータを書き込み、または取りに行く。二人は直接繋がらないため、同じ板の住人同士のIPは完全に隠蔽される。Bさんは二人のIPを知るが、データは暗号化されているため「この二人が何目的で自分（のハッシュID）に通信してきているのか」は確信できない。

## 4. Bootstrapping & Signaling (疎結合メッシュの構築)

ブラウザ単体では初期の接続を確立できないため、以下のフェーズを経て自律的なネットワーク（メッシュ）を構築・維持する。

### 4.1 初期の出会い（運営の軽量WebSocketトラッカー）
サイト（`reiwa-2ch.net`等）をホストする運営のWebサーバー内に、軽量なシグナリング用WebSocketサーバーを併設する。
- **役割**: 「待ち合わせ用の簡素な掲示板」。過去ログやメッセージの内容は一切中継・保存しない。
- **即時切断ルール**: 新規ユーザーがアクセスすると、トラッカーは「現在繋がっているランダムな数名（例: 6人）」のSDP（接続情報）を仲介する。6人とWebRTCが繋がった瞬間、**トラッカーとのWebSocket接続は意図的に切断（または極めて低い頻度のポーリングへ移行）**する。これによりサーバーへの依存と負担を回避する。

### 4.2 WebRTC越しのシグナリング (Peer Exchange / PEX)
トラッカーから離脱した後は、WebRTCのデータチャネル自体をシグナリング回線として利用する。
- 繋がっている隣のノードに対して「誰か新しく紹介して」とお願いし、P2Pネットワーク越しに見知らぬPeerのSDPを受け取って接続を拡張する。

### 4.3 接続度（Degree）の自律調整と撹拌
ネットワークが特定の島に分断されるのを防ぎ、局所的な囲い込みをなくすためのクライアントルール。
- **Min/Maxの維持**: 常に接続数（Degree）が4〜8人の範囲に収まるよう、相手がタブを閉じて切断されるたびにPEXで新規ノードを自動補充する。
- **ランダムウォーク撹拌**: 定期的に既存の接続を1つ切り、PEXで得た「友達の友達」と新しく結線し直すことで、全体が巨大な一つの「アメーバ状の網」として混ざり続けるようにする。

## 5. 暗号化モデル（URL埋め込み鍵 + Blind Mailbox）

### 5.1 Mega.nz方式の鍵共有
スレッド（または板）の共通鍵を **URLのフラグメント（`#` 以降）** に埋め込む。
`#` 以降はHTTPリクエストに含まれないため、運営のWebサーバー（トラッカー）にすら鍵は渡らない。

```
板のURL:
  https://reiwa-2ch.net/board/vip#boardkey=XXXX

スレッドの鍵（板鍵から自動派生）:
  thread_key = HMAC-SHA256(boardkey, "thread:12345")

→ 板のURLを知っていれば、その板の全スレッドを自動復号可能
```

### 5.2 Mailbox上のデータ状態
Mailbox担当ノード（中継役Bさん）が保管するデータは**全て暗号化済み**。

| Bさんが見えるもの | Bさんが見えないもの |
| :--- | :--- |
| `topic_hash` (不透明なハッシュ値) | 何の板・スレッドか（ハッシュの逆算不可能） |
| 暗号化されたBlob | レスの内容（復号鍵を知らない） |
| データのサイズ・タイムスタンプ | 誰が書いたか（署名も暗号化の内側） |

### 5.3 鍵の階層構造

```
[boardkey] (板ごとに1つ。URLフラグメントに埋め込み)
    │
    ├── thread_key = HMAC(boardkey, "thread:12345")
    │     └── レスの暗号化/復号に使用
    │
    ├── topic_hash = SHA256(thread_key)
    │     └── DHTでMailboxを探すための検索キー
    │         （thread_keyを知らないと計算できない）
    │
    └── gossip_tag = SHA256("gossip:" + topic_hash)[0:8]
          └── ゴシップ購読の識別タグ（8バイト）
              （topic_hashすら秘匿し、短縮ハッシュで購読する）
```

## 6. ゴシップ配布プロトコル: Broadcast Veil（購読宣言の完全廃止）

### 6.1 課題: 購読の宣言自体がIPと興味を紐付ける
標準のGossipSub（Pub/Sub）やBlind Subscription（gossip_tagの宣言）であっても、
隣接ノードに「何かを購読している」という事実を伝えた時点で、
同じタグを購読している複数ユーザーのIPが紐付けられてしまう。

### 6.2 解決: Broadcast Veil（全員が全パケットを転送、自分宛てかは自分だけが判断）
AETHERの `discovery_protocol_design.md` 案E「Broadcast Veil（全員が受信者）」を採用。
**「購読」という概念をプロトコルから完全に排除する。**

- 全ノードは受け取った全パケットを、中身を見ずに隣人へ転送する（Blind Flood）。
- 各ノードは受け取ったパケットに対して、自分が持っている全ての `thread_key` で復号を試みる。
- 復号に成功したら「自分宛て」と判断し、UIに表示する。
- 復号に失敗したら「自分には関係ないパケット」と判断し、何もしない（転送だけ）。

**→ 隣接ノードに対して「何を見ているか」を一切伝えないため、IPと興味の紐付けが構造的に不可能。**

### 6.3 Dandelion++ Stem（オプション: ON/OFF切替可能）
Broadcast Veilは「何を見ているか」を隠すが、「誰が書いたか（＝最初にパケットを発信したIP）」は
隣接ノードに `hop_count = 0` を見られることで特定される（相関攻撃）。
これを防ぐため、Moneroで実績のある **Dandelion++（タンポポ方式）** をオプション機能として導入する。

**ユーザー側のUI**: 設定画面に「🛡️ 匿名強化モード（Dandelion）」トグルスイッチ。
デフォルト: OFF（通常のBroadcast Veil）。ON にすると書き込み時にStemフェーズが挿入される。

#### 2フェーズ方式

```
Phase 1: Stem（茎）— 秘密の一本道
  パケットを全員にばら撒かず、隣接ノードから「1人だけ」をランダムに選んで渡す。
  受け取ったノードも同様に「1人だけ」に渡す。これを2〜4回繰り返す。

  A(作者) → B → C → D（ここまで秘密の一本道。Broadcastしない）

Phase 2: Fluff（綿毛）— 通常のBroadcast Veil
  最後のStemノード(D)が「ここからBroadcastする」と判断し、
  通常通り隣接全員にBlind Flood開始。hop_count = 0 をここでセット。

  結果: ネットワーク上では「Dが発信元」に見え、本当の作者Aは完全に匿名。
```

#### Stemパケットの識別防止（Plausible Deniability）
```
各ノードは、自分の書き込み以外にも他人のStemパケットを中継する。
これにより、攻撃者が最初のStemノードだったとしても:

  攻撃者が見える光景:
    「AのIPから、1分間に3本のStemパケットが来た」
     → 1本: A自身の書き込み
     → 2本: 他人の書き込みをAが中継しただけ
     → どれが本物か分からない（1/3の確率でしか当てられない）
```

#### 耐性（シミュレーション実証済み）
| 攻撃者支配率 | Dandelionなし | Dandelion++ ON |
| :---: | :---: | :---: |
| 10% | 47% 特定 | **3.3%** 特定 |
| 30% | 88% 特定 | **9.9%** 特定 |
| 50% | 98% 特定 | **16.6%** 特定 |
| 80% | 100% 特定 | **26.7%** 特定 |

**→ 攻撃者がネットワークの50%を支配しても、特定率はわずか17%。**

#### 遅延への影響
```
Stemの追加遅延: 3ホップ × 40ms = +120ms
合計: 通常の Broadcast Veil (0.3秒到達) + Stem (0.12秒) = 約0.42秒
→ 体感にはほぼ影響なし
```

### 6.4 書き込み（Publish）の伝搬フロー

```
[Dandelion OFF の場合]

1. ユーザーAが「キタ━━━(゜∀゜)━━━!!」と書き込む

2. ブラウザ内で以下を生成:
   plaintext  = { author: pubkey_hash, content: "キタ━━...", ts: 1710926715 }
   encrypted  = ChaCha20(thread_key, plaintext)
   signed     = Ed25519Sign(user_secret_key, encrypted)

3. ゴシップパケットを構成:
   {
     payload:    signed,              ← 暗号化済み（トピック情報も暗号の内側）
     pow_nonce:  987654321,           ← PoW証明
     hop_count:  0,                   ← TTL（中継回数制限）
     packet_id:  SHA256(payload)      ← 重複排除用
   }

4. 隣接する6人にパケットを送信（Broadcast開始）


[Dandelion ON の場合]

1〜2. 同上
3. Stemパケットを構成:
   {
     payload:    signed,
     pow_nonce:  987654321,
     stem_flag:  true,               ← Stemモード
     stem_ttl:   random(2, 4),       ← 残りのStemホップ数
     packet_id:  SHA256(payload)
   }

4. 隣接6人から「1人だけ」をランダムに選んで送信（Stem開始）
5. 受信ノードは stem_ttl-- して次の1人に転送
6. stem_ttl = 0 になったノードが stem_flag を除去、hop_count = 0 をセット
   → 通常のBroadcast Veil に移行（全隣接ノードにFlood）
```

### 6.5 中継ノード（隣人Bさん）の処理

```
Bさんがパケットを受け取った時:

1. stem_flag あり → Stemパケット
   - stem_ttl > 0: さらに「1人だけ」に転送 (stem_ttl--)
   - stem_ttl = 0: stem_flag 除去 → Fluff（Broadcast Veil）に移行
2. stem_flag なし → 通常のBroadcast Veilパケット
   - packet_id で重複チェック → 既に中継済みなら無視
   - hop_count チェック → 上限を超えていたら伝搬停止
   - PoW検証 → 不正なら即ドロップ
   - 自分が持っている全ての thread_key で復号を試みる（こっそり）
     - 成功 → 自分宛て！IndexedDBに保存 & UIに表示
     - 失敗 → 自分には関係ないパケット
   - hop_count++ して隣接ノード全員に転送（Blind Flood）

→ Bさんは「何かのパケットを中継した」ことしか分からない。
   誰が何を購読しているかという情報はプロトコル上に一切存在しない。
```

### 6.6 Stem切断対策（エコーリトライ + 冗長Stem）
Stemフェーズ中にノードがタブを閉じたり回線断が起きるとパケットがロストする問題への対策。

#### エコーリトライ方式
```
1. 作者AがStemパケットを送信
2. 同時に5秒のリトライタイマーを起動
3. パケットがFluff（Broadcast）に到達すると、Broadcast Veilで全ノードに届く
   → 作者Aにも自分のパケットが戻ってくる（エコー）

   [5秒以内にエコー確認] → 成功。タイマー解除。
   [5秒経過、エコーなし] → Stemで消失と判断。別の隣人で再送。
   [2回目も失敗]         → Dandelion OFF にフォールバック（直接Broadcast）

結果: 最悪でも15秒以内に投稿が反映される。パケットが消えることは絶対にない。
```

#### 冗長Stem（ブラックホール攻撃対策）
```
Stemパケットを1本ではなく2本の独立経路で同時に送信:

  A(作者) → B1 → C1 → D1(Fluff)    ← 経路1
  A(作者) → B2 → C2 → D2(Fluff)    ← 経路2

攻撃者20%の場合:
  1本Stem ロスト率: 48.8%
  2本Stem ロスト率: 23.8%（両方が同時にドロップされる確率）
  2本Stem + リトライ: → ほぼ0%
```
匿名性への影響: 2本に増えても、各経路は独立した別のノードを通るため、
攻撃者が両方の経路の初段を同時に支配する確率は p² と低く、匿名性は十分維持される。

### 6.7 帯域保護（3段階スロットル + パケットサイズ上限）
Broadcast Veilでは全パケットが全ノードに届くため、トラフィック増大時のユーザー保護が必要。

#### パケットサイズ上限（ハードリミット）
```
最大ペイロードサイズ: 2KB/パケット（暗号化後）
→ 2KBを超えるパケットは中継ノードが問答無用でドロップ
→ 巨大AAやBase64画像の埋め込みを構造的に不可能にする
→ 通常のテキスト投稿（数百B〜1KB）には十分なサイズ
```

#### PoW難易度のサイズ連動
```
difficulty = BASE_DIFFICULTY + log2(payload_size / 256)

  256Bのパケット: 基準難易度（~0.5秒）
  1KBのパケット:  基準 + 2（~2秒）
  2KBのパケット:  基準 + 3（~4秒）

→ 大きいパケットを大量に投げるのは計算コスト的に非常に高くなる
```

#### ノード別の帯域予算（ユーザーが設定可能）
```
設定UI（🔧 通信量設定）:

  🎬 高画質モード（デフォルト）: 625 KB/s   ← 1080p動画配信と同等
  📺 標準モード:                200 KB/s   ← SD動画相当
  📱 ギガ節約モード:             50 KB/s   ← 音声通話相当

予算を超えた分のパケットはランダムにドロップ（古いものから優先的に破棄）。

月間データ量（8時間/日利用）:
  高画質 625KB/s:  ~483 GB/月（光回線・Wi-Fiユーザー向け）
  標準   200KB/s:  ~155 GB/月
  ギガ節約 50KB/s:  ~39 GB/月（格安SIM 50GBプランでも余裕）
```

#### 実際のトラフィック vs 帯域予算
```
通常運用（50件/秒 × 500B = 25KB/s）:
  → ギガ節約モード（50KB/s）でも全パケット受信可能 ✅

炎上時（200件/秒 × 2KB = 400KB/s）:
  → 高画質モード（625KB/s）: 全パケット受信 ✅
  → 標準モード（200KB/s）: 半分ドロップ。閲覧は可能だが一部欠落 ⚠️
  → ギガ節約（50KB/s）: 大幅にドロップ。Mailboxから後で同期すればOK ⚠️

パケ代テロ（500件/秒 × 5KB）:
  → 2KBキャップで大半は中継ノードが弾く → テロ自体が成立しない ✅
```

### 6.8 復号負荷の最適化（バッテリー枯渇攻撃対策）
Broadcast Veilでは全パケットに対して全 `thread_key` で復号を試みるため、
購読スレッド数が多いと復号コストがバッテリーを圧迫する。
攻撃者が大量のゴミパケットを投入して全ノードに無駄な復号を強制する攻撃も想定される。

#### マジックバイト部分復号フィルタ
暗号化の**内側**に4バイトの固定マジックナンバーを埋め込み、
先頭4バイトだけの超軽量な部分復号で「自分宛てか？」を判定する。

```
送信側:
  plaintext = "AETH" + actual_content        ← 先頭4バイトが固定マジック
  encrypted = ChaCha20(thread_key, nonce, plaintext)

受信側（各thread_keyについて）:
  1. 先頭4バイトだけを部分復号（ストリーム暗号なのでXOR 4回 = ナノ秒）
  2. 結果が "AETH" か？
     → 不一致 → スキップ（フル復号をしない）
     → 一致   → ヒット！残り全体を復号 → UIに表示

誤検知率: 1 / 2^32 ≈ 0.00000002%（事実上ゼロ）
```

#### コスト比較（20スレ購読、50パケット/秒）
```
改善前: 50 × 20 = 1,000回/秒 の ChaCha20 フル復号（各500B〜2KB）
        → 数百万バイトの暗号処理/秒 → バッテリー直撃

改善後: 50 × 20 = 1,000回/秒 の 4バイトXOR（部分復号）
        → 4,000バイトの演算/秒 → 実質ゼロコスト
        + ヒットした数件だけフル復号
        → 99%以上の復号処理をスキップ
```

#### プライバシーへの影響: **ゼロ**
マジックバイトは暗号化の内側に埋め込まれており、パケットの外側（平文部分）には何も追加しない。
`thread_key` を知らない中継ノードにはマジックバイトの存在自体が見えないため、
**Broadcast Veil のプライバシーは完全に維持される**。

### 6.9 確率的キャッシュ拡散（Probabilistic Cache Seeding）
Broadcast Veilの帯域コストを70%削減し、スケール限界を10倍以上に拡張する最適化。

#### 基本原理
全パケットの本文を全ノードに送る代わりに、**通知（50B）だけを全員に送り、
本文（500B〜2KB）はメッシュの20%にだけキャッシュさせる**。
購読者は自分の隣人からキャッシュを取得する（Pull）。

#### データフロー
```
1. 書き込み者が投稿
2. パケットをBroadcast Veilで送信する際:
   ・通知ヘッダー（50B）: 全隣人に必ず転送（従来通り）
   ・本文コンテンツ: 確率 p=20% でキャッシュ（ローカルに60秒間保持）

3. 各中継ノード:
   ・通知は必ず転送（Broadcast Veil）
   ・20%の確率でコンテンツもキャッシュ
   → 10,000人ネットワークなら2,000人がキャッシュ保持

4. 購読者ノード:
   ・通知を受信 → マジックバイトで「自分宛て」と判定
   ・隣人に「packet_hash X のデータある？」と問い合わせ
   ・隣人6人のうち1人以上がキャッシュしている確率: 74%（1hop）
   ・2hop以内に見つかる確率: 99.97%
```

#### ヒット率（10万回試行で検証済み、ノード数に依存しない）
```
p=20%, degree=6:
  1hop ヒット: 73.87%（隣人から即取得、20ms）
  2hop ヒット: 99.97%（隣人の隣人から取得、40ms）
  失敗率:      0.03%
```

#### Pull失敗時のリトライ
```
1回目: 隣人に問い合わせ（1-2hop）              → 99.97% 成功
失敗（0.03%）
  ↓ 200ms待機（キャッシュ伝播の猶予）
2回目: 別の隣人に問い合わせ                    → 99.97% 成功
失敗（0.03% × 0.03% = 0.0000009%）
  ↓
3回目: DHT Mailbox にフォールバック             → 100% 成功

最終的なパケットロスト率: 0%（Mailboxが最終保証）
最悪ケースの追加遅延: 400ms（リトライ200ms × 2回）
```

#### 帯域の比較（50件/秒、500B/件）
```
Full Broadcast（従来）:    25 KB/s（全ノード）
Cache p=20% + Pull:         7 KB/s（全ノード）← 72%削減

帯域内訳:
  通知（50B × 50件/秒）:       2.5 KB/s（全ノードが受信）
  キャッシュ（500B × 50 × 20%）: 4.9 KB/s（20%のノードだけ）
  Pull応答:                     微小（隣人からの1-2件）
```

#### スケール限界の拡張
```
1080p帯域（625KB/s）で耐えられる投稿頻度:
  Full Broadcast: 秒間1,250件 → 同接 ~125万人
  Cache p=20%:    秒間3,657件 → 同接 ~365万人（2ch全盛期の36倍）
```

#### Pull負荷の分散（ホットスポットなし）
```
DHT Pull（K=5に集中）:
  1万人同時Pull → 各DHTノード 2,000回/s → ❌ 破綻

Cache Pull（p=20%に分散）:
  1万人同時Pull → 各キャッシュ 0.5回/s → ✅ 余裕
  100万人中10万人Pull → 各キャッシュ 0.5回/s → ✅ 余裕
  → ネットワークが大きくなるほどキャッシュ保持者も増えるため、
    Pull負荷はむしろ下がる
```

#### ⚠️ プライバシーへの影響と致命的脆弱性 (Vulnerability B)
```
新たに漏れうる情報:
  隣り合うノードに「packet_hash X のデータが欲しい」と問い合わせる行為（Pull要求）

致命的な交差攻撃（Intersection Attack）のリスク:
  1. 攻撃者が公開板の `boardkey` を所持している場合、ネットワークに流れる「通知パケット」を復号できる。
  2. 攻撃者は「packet_hash X = VIP板の特定の投稿」と完璧に特定できる。
  3. その直後、隣のターゲットノードが「packet_hash X のデータある？」とPullしてきた場合。
  4. 攻撃者から見て「このIPのユーザーはVIP板の住人だ」と完全に確定する。

評価（棄却判定）:
  公開掲示板において、板の暗号鍵は攻撃者（ROM専）も保有している前提に立つべきである。そのため、特定データの取得要求を行うPull型の方式は、**「完璧な閲覧プライバシーの死守（完全な無関心）」**という Broadcast Veil の根幹を破壊する。
  スケーラビリティには極めて優れるが、この「閲覧バレ」の致命的脆弱性に対抗できないため、AETHERのコアプロトコルとしては【不採用（棄却）】とする。
```

### 6.10 サブメッシュ分割による超大規模化（K-Anonymous Zone Routing）

Broadcast Veilの「完全な無関心（Pull不要の完全Push配信）」を維持したまま、ネットワークの動的な論理分割によって帯域コストを **93.75% 削減** し、スケーラビリティの限界を数百万人に押し上げる最終的なアーキテクチャ。

#### 基本原理: 巨大な「Zone」への機械的分割と自律配置
リアルタイム配信を行うPubSubメッシュ（Broadcast Veil）を、全世界で `1` つではなく、`256` 個の「Zone」に論理分割する。

1. **Zoneの決定**
   スレッドの暗号鍵（`thread_key`）を用いて、そのスレッドが所属するZoneを一意に決定する。
   `Zone_ID = SHA256(thread_key) mod 256`
   ※ これにより、VIP板も料理板も、1つのZoneあたり数百〜数千の全く無関係なスレが無作為に混在する。

2. **強制ダミー参加と「固定16枠」のルール**
   全クライアントは、自分の興味に関係なく「必ず合計 **16個** のZoneメッシュ」に同時参加する。
   - 自分が読みたいスレッドが Zone 12 と 45 にある場合。
   - クライアントは裏でランダムな 14個の「ダミーZone」を追加抽選する（例: 7, 12, 45, 53, ... 計16個）。
   - そして「私はこの16個のZoneをリッスンする」と宣言し、16個に絞られたBroadcast Veilを同時に受信する。

#### プライバシーへの影響（完全なる非対称性）
**【閲覧者の保護（K-匿名性）】**
- 攻撃者の視点: 「このIPは、Zone 12を含む16個のZoneを受信している」と気付く。
- 強い否認権A（ダミーの存在）: 「16個中15個はシステムが勝手に選んだダミーだ（攻撃者が本命だと断定できる確率はたったの 6.25%）」。
- 強い否認権B（K-Anonymity）: 「仮にZone 12が本命だとしても、Zone 12には他板のアニメスレも密集している。私はアニメスレの住人だ」。
- ⚠️ **自己防衛ルール**: 交差攻撃（Intersection Attack）を防ぐため、16個のZoneセットは**「IPアドレスまたはセッションが変わるまで絶対に再抽選してはいけない」**。固定を厳守すれば、どれだけ長期観測されてもダミーが剥がれ落ちず、永遠にターゲットを絞り込めない。

**【送信者の保護（Dandelion++ 越境インジェクション）】**
作者Aが「Zone 42」に書き込む場合、**A自身がZone 42のメッシュに繋ぐ必要はない**。
1. `zone_id: 42` を平文で付けた Stem パケットを作る。
2. 自分が今繋がっているランダムな隣接ノード（自分が所属する16個のZone関係なし）に単体転送する（Dandelion++ のStemフェーズ開始）。
3. バケツリレーの末にコイントスでFluff（拡散）を引き当てた赤の他人のノード（Fさん）が、自らDHT経由で Zone 42 の入り口を見つけ、遠隔地からパケットを投下（Inject）する。
4. Zone 42の住人には「Fさんが持ってきた」としか見えない。作者Aの痕跡は完全に消滅する。

#### スケール限界と帯域の破壊的最適化（シミュレーション実証済み）

| ネットワーク規模 | 1Zone辺りの人数 | リアルタイムメッシュの挙動 | 1ノード負荷 |
|:---|:---:|:---|:---:|
| **過疎期（10人）** | 平均 0.6人 | 全てのZoneの約53%は「無人Zone」となる。無人Zone宛のパケットはPush配信がスキップされ、**直接 DHT（Mailbox）に格納**される。 | 0 KB/s |
| **成長期（1 千人）** | 平均 62人 | `libp2p-gossipsub` が完璧な木構造メッシュを形成し始める。「空のZone」は確率0%となり、全256Zoneで安定したリアルタイム配信が稼働。 | 約 0.3 KB/s |
| **全盛期（100万人）** | 平均 6.2万人 | 全Zoneが6万人規模の強固なメッシュとなる。 | **約 15.3 KB/s** |

* **帯域削減率**: 全体の 16 / 256 しか受信しないため、**93.75%カット（1/16）**。
* **アーキテクチャの限界点**: 100万人（秒間5000件=2ch全盛期のピーク以上）が同時接続しても、1ノードの帯域負荷は「驚異の 15 KB/s」に収まる。純粋なP2Pアーキテクチャ単体で数百万〜数千万人のスケーラビリティを発揮する。

## 7. データ永続化層（Distributed Browser Mailbox）

リアルタイムのBroadcast Veilと、永続化のMailboxの**2層構造**。

| 層 | 方式 | 用途 |
|:---|:---|:---|
| **リアルタイム層** | Broadcast Veil（全員転送、自分だけ判定） | 新しいレスの即時配信 |
| **永続化層** | Kademlia DHT + K-Replication Mailbox | 過去ログの保管と新規参加者への同期 |

### 7.1 DHTベースのMailbox割り当て（K-Replication）
`topic_hash` に最も近いNode IDを持つ **上位K人（K=5）** が、
そのスレッドの過去ログの暗号化Blobを `IndexedDB` に保管する義務を負う。
5人中3人が生存していれば完全なデータ復元が可能（Reed-Solomon 3+2）。

### 7.2 ゼロノードからの復元 (Synchronization)
新たにスレッドを開いたユーザーは、DHTを経由してMailbox担当ノード（K=5のうち誰か）を発見し、
暗号化された過去ログBlobを取得。自身のブラウザ内で `thread_key` を使って復号し、画面に描画する。

## 8. 板・スレッドの検索と発見（Discovery Model）

### 8.1 ハイブリッド・ディスカバリー
「プライバシー（暗号で閉じる）」と「ディスカバラビリティ（みんなに見つけてもらう）」のトレードオフを、
**板と スレッドで責任を分離する**ことで解決する。

| レイヤー | 発見方法 | 管理者 |
| :--- | :--- | :--- |
| **板の発見** | 運営のWebサイト（HTML）に板一覧（URLとboardkey付き）を公開 | 運営 |
| **スレッドの発見** | P2P（DHT上の暗号化メタデータ）で分散管理 | ユーザー全員 |

- **板一覧は公開情報**: 2chの「板一覧」と同じ。サイトのトップページに板のURL（boardkey入り）が並ぶ。
- **スレッド一覧は分散情報**: 板のMailbox（DHT上）に暗号化されたスレッドメタデータが保管される。boardkeyを知っている人だけがスレッド一覧を閲覧できる。
- **板のboardkeyが拡散していれば、運営のサイトが落ちても板は存続できる**（URLを知っている人がいる限り）。

### 8.2 スレッドの公開/非公開モード（UXで切り替え可能）

スレッド作成時に、スレ主が「公開」or「非公開（秘密）」を選択できる。

#### 公開スレッド（デフォルト）
```
1. スレ主が「スレを立てる」ボタンを押す
2. スレッドのメタデータ（タイトル、作成日時、thread_id）を boardkey で暗号化
3. 板の topic_hash に紐付く Mailbox に PUT（スレッド一覧に登録）
4. 板のURLを知っている全員がスレッド一覧からこのスレッドを発見できる

→ 通常の2chのスレッドと同じ。板を見れば誰でも見つけられる。
```

#### 非公開スレッド（「秘密スレ」にチェック）
```
1. スレ主が「スレを立てる」ボタンを押す（「秘密スレ」ON）
2. Mailbox への メタデータ登録をスキップ（スレッド一覧に載らない）
3. スレッドのURL（thread_key入り）を直接知っている人だけがアクセス可能
4. URLはDMやSNSで個別に共有する

→ YouTubeの「限定公開」と同じ概念。
   URLを知らない人からは存在自体が見えない。
```

#### 暗号的な違い
```
公開スレッド:
  thread_key = HMAC(boardkey, "thread:12345")
  → boardkey から自動派生。板を知っていれば読める。

非公開スレッド:
  thread_key = ランダム生成（boardkeyと無関係）
  → URLに thread_key を直接埋め込む。板の鍵では復号不可能。
  → https://reiwa-2ch.net/secret#key=ZZZZZZ

結果: 同じ板の公開スレ住人が、存在すら知らない秘密の部屋を同じネットワーク上に持てる。
```

### 8.3 Web検索インデックス（公開スレッドの自動登録）
公開スレッドの「発見しやすさ」を最大化するため、運営のWebサーバーに軽量な検索APIを持たせる。

#### スレ立て時の自動登録フロー
```
スレ主が「公開スレ」を立てた瞬間、ブラウザが2つの処理を同時に実行:

a) P2P側（従来通り）:
   暗号化メタデータを板のMailboxにPUT

b) Web側:
   スレタイ・板名・>>1の本文・作成日時をAPIで運営サーバーに送信
   POST /api/threads {
     board: "vip",
     title: "キタ━━━!!",
     thread_id: "12345",
     op_content: "VIPからき☆すた",    ← >>1の本文（公開スレなので平文でOK）
     created_at: ...
   }
```

#### 運営サーバーが保持するもの / 保持しないもの
| 保持する（検索インデックス） | **絶対に保持しない** |
| :--- | :--- |
| スレッドのタイトル | >>2以降のレス内容 |
| >>1の本文（公開スレのみ） | ユーザーのIPアドレス |
| 所属する板名・作成日時 | 書き込み者の公開鍵・署名 |
| thread_id（スレッドの識別子） | boardkey / thread_key（復号鍵） |

**→ サーバーは「電話帳（スレタイと>>1だけ）」を持ち、「通話の中身（>>2以降）」には一切触れない。**

#### サイト内検索UI
```
┌──────────────────────────────────────────────┐
│  🔍 スレッド検索 [キタ━━━          ] [検索]   │
├──────────────────────────────────────────────┤
│  VIP板 | キタ━━━(゜∀゜)━━━!!     | 3分前    │
│  VIP板 | キタ━━━ス レ           | 1時間前  │
│  なんJ | キタ━━━打線組んだ       | 2時間前  │
├──────────────────────────────────────────────┤
│  クリック → P2P接続開始 → Mailboxから過去ログ取得│
│  → thread_key で復号 → スレッド表示            │
└──────────────────────────────────────────────┘
```

#### 検閲耐性
- 運営がサーバーの検索インデックスから特定のスレッドを消しても、
  **P2P側（Mailbox）にメタデータが残っている限り、板のURLを知っている人はスレッドにアクセス可能**。
- 検索インデックスはあくまで「便利な入り口」であり、P2Pの暗号化世界の一次ソースではない。

### 8.4 Google検索対応（SSRプレビューページ）
公開スレッドをGoogle検索からも発見できるようにする。

#### 仕組み: サーバーサイドレンダリング（SSR）
スレッドが公開登録された際、Webサーバーがクローラー向けの静的HTMLを自動生成する。

```
生成されるHTML:

    <meta property="og:title" content="キタ━━━(゜∀゜)━━━!!">
    <meta property="og:description" content="VIPからき☆すた">
  </head>
  <body>
    <h1>キタ━━━(゜∀゜)━━━!!</h1>
    <article>
      <p>1: VIPからき☆すた</p>
    </article>
    <p>このスレッドの続きを読むにはP2Pネットワークに接続してください。</p>
    <script>
      // 公開板の boardkey はJS内に埋め込み（板一覧と同じ公開情報）
      // P2P接続を開始して >>2以降のレスをロード...
    </script>
  </body>

```

#### ユーザーの体験フロー

#### セキュリティ上の影響
| 対象 | Googleに載るか | 理由 |
| :--- | :---: | :--- |
| 公開スレッドの **タイトルと>>1** | ✅ 載る | サーバーに平文で登録済み。公開情報。 |
| 公開スレッドの **>>2以降** | ❌ 載らない | P2Pの暗号化世界にしか存在しない。 |
| **秘密スレッド** | ❌ 一切載らない | サーバーにデータを送信しない。 |
| **boardkey** | ❌ 載らない | URLフラグメント(#)はクロール対象外。 |

## 9. 匿名アイデンティティとスパム対策

### 9.1 Ed25519による署名 (Nostrライク)
- **ID（その日限り）**: スレッド参加時に一時生成される鍵ペア。IPではなく公開鍵のハッシュで発言者を一意に識別。
- **トリップ（永続）**: `IndexedDB` に永続保存させた任意の秘密鍵。これで署名することで、IPが変動しても厳密に本人証明が可能。

### 9.2 WASM Client-Side PoW (Proof of Work)
シビル攻撃（ボットによる大量スパム）を防ぐため、書き込み（送信）ボタンを押す前に、ブラウザ上の **WebAssembly (Argon2等)** で数秒かかるハッシュ計算を強制する。

### 9.3 PoW難易度の自律合意形成（Bitcoin簡易版）
中央管理者なしで全ノードが「正しい難易度」を自律的に算出する仕組み。

#### アルゴリズム
```
入力: 自分のIndexedDBに保存されている「直近 WINDOW 件のレス」のタイムスタンプ

パラメータ:
  WINDOW          = 100       # 直近何件を見るか
  TARGET_INTERVAL = 3000ms    # 理想的な投稿間隔 = 3秒に1件
  BASE_DIFFICULTY = 12        # 基準難易度（先頭12ビットがゼロ）
  MIN_DIFFICULTY  = 8         # 最低難易度
  MAX_DIFFICULTY  = 24        # 最大難易度

計算:
  1. elapsed_ms = 最新レスの時刻 - 100件前のレスの時刻
  2. actual_interval = elapsed_ms / WINDOW
  3. ratio = TARGET_INTERVAL / actual_interval
  4. difficulty = clamp(BASE_DIFFICULTY + log2(ratio), MIN, MAX)
```

#### 合意の仕組み（なぜ全員が同じ値に収束するか）
```
Broadcast Veil により全パケットが全ノードに配信される（100%配信率が実証済み）
→ 全員が「同じ100件のレス」のタイムスタンプを保持
→ 全員が同じ計算式を独立に実行
→ 全員が同じ difficulty を算出

スパマーが「難易度は8だよ」と嘘の低い難易度でパケットを投げた場合:
→ 正直なノードは自分で計算した difficulty = 15 で検証
→ 15ビットを満たさないPoWナンスのパケットは即ドロップ
→ スパムはネットワークを通過できない
```

#### 難易度と体感の対応表
| 状況 | 実際の間隔 | 算出難易度 | 送信側の計算時間 |
| :--- | :--- | :--- | :--- |
| 過疎（1時間に数件） | 36000ms | 8（最低） | ~0.1秒 |
| 平和（3秒に1件） | 3000ms | 12 | ~0.5秒 |
| 盛り上がり中 | 300ms | 15 | ~3秒 |
| 爆撃・炎上中 | 30ms | 18 | ~30秒（事実上の書き込み制限） |

#### 新規参加者の対応
過去ログが0件の場合は `MIN_DIFFICULTY = 8` をデフォルトとして使用。
Mailboxからの過去ログ同期（約0.7秒）が完了した時点で、他のノードと同じ難易度に収束する。

## 10. パケット構造定義（3層構造）

ネットワーク上を流れるゴシップパケットは、**外側（平文）・暗号化層・署名層**の3層で構成される。

### 10.1 パケット全体像

```
┌─────────────────────────────────────────────────────┐
│ 外側（平文）: 中継ノードが見える部分                    │
│                                                       │
│  packet_id      : [32 bytes]  SHA256(payload) 重複排除 │
│  hop_count      : u8          TTL（0〜30）             │
│  pow_nonce      : u64         PoW証明ナンス             │
│  pow_difficulty  : u8          計算時の難易度             │
│  timestamp      : u64         送信時刻 (UNIX ms)       │
│  payload_size   : u32         暗号化データのサイズ        │
│  payload        : [bytes]     暗号化済みデータ本体       │
│                                                       │
├─────────────────────────────────────────────────────┤
│ 暗号化層（thread_keyで暗号化。URLを知る人だけ復号可）    │
│                                                       │
│  board_id       : string      板名 ("vip")             │
│  thread_id      : string      スレッドID ("12345")     │
│  post_type      : enum        "post" | "thread_meta"   │
│  inner_payload  : [bytes]     署名済みデータ             │
│                                                       │
├─────────────────────────────────────────────────────┤
│ 署名層（Ed25519署名で改竄防止）                         │
│                                                       │
│  author_pubkey  : [32 bytes]  書き込み者の公開鍵        │
│  content        : string      本文                      │
│  post_number    : u32         レス番号                   │
│  reply_to       : u32 | null  アンカー先                 │
│  created_at     : u64         作成時刻 (UNIX ms)        │
│  trip_signature : [64 bytes]  トリップ署名（任意）       │
│  signature      : [64 bytes]  Ed25519(上記全フィールド)  │
│                                                       │
└─────────────────────────────────────────────────────┘
```

### 10.2 各層の可視性

| 層 | 見える人 | 目的 |
| :--- | :--- | :--- |
| **外側（平文）** | 全ノード（中継者含む） | 中継判断、PoW検証、重複排除、TTL管理 |
| **暗号化層** | URLを知る人だけ | トピック特定、投稿種別の判別 |
| **署名層** | URLを知る人だけ | 本文閲覧、投稿者の認証、改竄検知 |

### 10.3 中継ノードが行える処理 / 行えない処理

| 中継ノードが**できること** | 中継ノードが**できないこと** |
| :--- | :--- |
| packet_id で重複排除 | 何の板・スレッドか知ること |
| hop_count でTTL管理 | レスの内容を読むこと |
| pow_nonce でスパム検証 | 書き込み者を特定すること |
| payload_size でサイズ制限 | パケットを改竄すること（署名検証で即バレ） |

### 10.4 投稿種別 (post_type)

| 種別 | 用途 | 備考 |
| :--- | :--- | :--- |
| `post` | 通常のレス書き込み | スレッド内の発言 |
| `thread_meta` | スレッド作成時のメタデータ | タイトル、作成日時。板のMailboxに登録される |

## 11. NAT越え戦略（TURNサーバー不要）

### 11.1 WebRTC + STUN で十分な理由
WebRTCのICEフレームワークは、無料のSTUNサーバーを使ってNAT越えを自動的に処理する。

| NATの種類 | STUN成功率 | 主な環境 |
| :--- | :--- | :--- |
| Full Cone | ✅ 100% | 安いルーター |
| Address-Restricted | ✅ 成功 | 一般的な家庭ルーター |
| Port-Restricted | ✅ 成功 | モバイルキャリア（CGNAT） |
| Symmetric | ❌ 失敗 | 企業FW、ホテルWiFi |

**スマホのモバイル回線（docomo, au, SoftBank, 楽天）はほぼ全てPort-Restricted以下であり、STUNで突破可能。**

Google統計（2024年）によるSTUNのみでの接続成功率: **約85〜92%**。
失敗する8〜15%の大半は企業ネットワークや学校のファイアウォール。

### 11.2 使用するSTUNサーバー（全て無料・公開）
```
stun:stun.l.google.com:19302       (Google)
stun:stun.cloudflare.com:3478      (Cloudflare)
stun:stun.services.mozilla.com     (Mozilla)
```

### 11.3 繋がらない場合のUX
Symmetric NAT環境（企業PC等）のユーザーには以下を表示:
「あなたのネットワーク環境ではP2P接続できません。Wi-Fiまたはモバイル回線でお試しください」

## 12. バックグラウンドタブ問題の対処

### 12.1 ブラウザの制限と WebRTC の特権
```
バックグラウンドタブの制限（Chrome/Firefox/Safari 2025年時点）:
  setTimeout/setInterval → 最大 1回/分 に制限
  requestAnimationFrame → 完全停止
  WebSocket             → 制限なし（動く）
  WebRTC DataChannel    → ★制限なし★（独立スレッドで動作）
```
**WebRTCのDataChannelはスロットリング対象外**。接続は維持され、パケットの送受信は継続される。

### 12.2 Web Worker によるJS処理の分離
パケットの中継・復号・IndexedDB書き込みなどのJS処理をWeb Workerに委譲する。
Web Workerはバックグラウンドでも制限されない（Chrome/Firefox共通）。

```
メインスレッド（制限される）:
  → UI描画のみ

Web Worker（制限されない）:
  ├── パケットの受信・送信処理
  ├── ChaCha20 暗号化/復号
  ├── PoW検証
  ├── ゴシップの中継判断（Broadcast Veil）
  └── IndexedDB（Mailboxデータ）への書き込み
```
Discord, Slack, Google Meet も同じ手法でバックグラウンド通信を維持している。

## 13. シビル攻撃・IP収集攻撃への防御（4層防御）

攻撃者がHeadless Chromeを大量起動し、ネットワークの過半数を支配しようとする攻撃への対策。

### 13.1 第1層: Cloudflare Bot Protection（トラッカーの門番）
トラッカー（WebSocketシグナリングサーバー）はCloudflareの背後に配置する。

```
接続フロー:
1. ユーザーがサイトを開く
   → Cloudflare Bot Fight Mode が自動でブラウザフィンガープリント検証
   → Headless Chrome / Puppeteer / curl / スクリプト → 即ブロック

2. 人間と判定 → Cloudflare Turnstile トークンを自動取得
   （ユーザーには見えない。画像を選ぶCAPTCHAは不要）

3. WebSocket(トラッカー) 接続時に Turnstile トークンをヘッダーに含める

4. トラッカー側で Cloudflare Turnstile API にトークンを検証
   → 有効 → ピアリスト(6人)を紹介
   → 無効 → 接続拒否

5. 以降は P2P（WebRTC）のみ。Cloudflare は一切関与しない。
```

**コスト: 無料**（Cloudflare Free プランに Bot Fight Mode と Turnstile が含まれる）
**効果: ボットの大量参加を事実上不可能にする**（Turnstileの突破コストは1リクエストあたり数円〜数十円）

### 13.2 第2層: トラッカーのレートリミット
Cloudflareを突破された場合のバックアップ。

```
- 同一IPからのWebSocket接続: 30秒に1回まで
- 1回の接続で紹介するピア: 最大6人（全体リストは渡さない）
- トラッカーのログ: メモリ上のみ保持、WebSocket切断と同時に即破棄

→ 仮にTurnstileを突破しても:
  1万ノードを立てるには 1万 × 30秒 = 83時間（3.5日間）必要
  しかもIP1つにつき30秒に1ノードしか参加できない
```

### 13.3 第3層: Node Aging（I2P方式の信頼スコア）
メッシュ参加後、ネットワーク内部で新参ノードの権限を制限する。

```
各ノードに「生存時間（Age）」を持たせる:
  Age < 10分  → 「新参者」
    - Mailbox担当に選ばれない（DHTのk-bucketで優先度最低）
    - PEXで他人に紹介されない（信頼できない人を広めない）
    - ゴシップの中継はできるが、パケットのドロップは検知対象

  Age ≥ 10分 → 「一般ノード」
    - 通常の全機能が使える

→ 攻撃者が1万ノードを一斉投入しても、
  最初の10分間はMailbox・DHTの操作ができないため、
  正直なノードが異常を検知して対応する猶予がある。
```

### 13.4 第4層: K-Replicationの多数決（データ改竄の最終防衛）
DHT Mailboxの担当者が複数（K=5）いることを活かした改竄検知。

```
ユーザーがMailboxからデータを取得する時:
  5ノード中3ノード以上が同じデータを返す → 正しいデータとして採用
  異なるデータを返すノード → 「嘘つき」としてブラックリスト登録

Node Aging と組み合わせ:
  Age が古い（信頼度の高い）ノードがMailbox担当に優先される
  → 攻撃者の新参ノードがMailboxを乗っ取るのは極めて困難
```

### 13.5 4層防御のまとめ

| 層 | 対策 | 参考元 | 防ぐもの |
| :--- | :--- | :--- | :--- |
| **第1層** | Cloudflare Bot Fight Mode + Turnstile | Cloudflare | ボットの自動大量参加 |
| **第2層** | トラッカーのレートリミット + ログ即破棄 | BitTorrent | IP収集の速度制限 |
| **第3層** | Node Aging（新参ノードの権限制限） | I2P | DHT/Mailboxの乗っ取り |
| **第4層** | K-Replication の多数決 | AETHER本体 | データの改竄・検閲 |

**根本防御（全層を貫く設計思想）:**
仮にIPが集められても、Broadcast Veil により「そのIPが何をしているか（どの板を見ているか）」は
構造的に不可視。**IPの収集が成功しても、得られる情報に価値がない。**
